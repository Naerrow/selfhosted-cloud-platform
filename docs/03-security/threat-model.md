# STRIDE 위협 모델 (v0)

> 본 문서는 STRIDE 분류 체계로 본 플랫폼의 위협을 도출·정리한다.
> **Phase 0 Exit 게이트** 산출물 v0이며, 매 Phase 종료 시 갱신한다 (ROADMAP 횡단 원칙).
>
> STRIDE 정의: [`../00-overview/glossary.md`](../00-overview/glossary.md) §D — Spoofing / Tampering / Repudiation / Information Disclosure / Denial of Service / Elevation of Privilege.

## 0. 방법론과 범위

- **입력 산출물**:
  - 신뢰 경계·데이터 플로우: [`../01-architecture/trust-boundaries.md`](../01-architecture/trust-boundaries.md) (TB-1~TB-7). 본 문서는 그 위협 시드를 정식 위협으로 전개한다.
  - 시스템 컨텍스트·컨테이너: [`../01-architecture/c4-level1-context.md`](../01-architecture/c4-level1-context.md), [`../01-architecture/c4-level2-containers.md`](../01-architecture/c4-level2-containers.md)
  - 공격 주체·권한 비대칭: [`../00-overview/personas.md`](../00-overview/personas.md) §7
  - 리스크 레지스터: [`../90-meta/risks.md`](../90-meta/risks.md) (특히 R-004·R-006·R-009·R-010)
- **방법**: 각 신뢰 경계(TB)와 핵심 자산을 STRIDE 6범주로 점검한다. 위협마다 ID(`TH-NN`), 대응 통제, 관련 ADR·리스크 매핑을 남긴다. (NFR ID 매핑은 차기 버전에서 추가 검토.)
- **v0의 한계 (정직성 고지)**:
  - ADR-0004(백엔드 스택)·**ADR-0005(멀티테넌시 격리)**·**ADR-0006(빌링 무결성)**은 **Accepted**. ADR-0007(시크릿 관리)는 아직 **Proposed**이므로 해당 영역(TB-7) 통제는 "확정 예정"으로 표기하며, Accepted 시 본 문서를 갱신한다.
  - v0는 **위협 식별과 통제 방향**까지를 목표로 한다. 정량 리스크 스코어링(DREAD 등)·공격 트리 상세화는 Phase 2(IAM·멀티테넌시) 진입 시 심화한다.
  - 본 문서는 외부 공개 추적 문서다 ([ADR-0009](../02-adr/0009-internal-only-design-docs.md)의 **내부 전용 산출물** 대상은 아님 — 즉 공개된다). 따라서 ADR-0009의 공개 문서 규칙(내부 전용 파일명·지침 직접 인용 금지)은 본 문서에도 적용되며, 통제 인용은 ADR·vision·scope 기준으로 한다.

## 1. 공격 주체 (Threat Actors)

| 주체 | 설명 | 핵심 가정 |
|---|---|---|
| **외부 익명 공격자** | 인증 없이 인터넷에서 접근 | TB-1 너머에서 인증·인가 우회를 시도 |
| **악의적 P-EU** | 정당한 엔드 유저 자격이 있으나 권한을 넘어서려는 자 | 같은/다른 테넌트 자원으로 권한 상승(→P-TA) 시도 |
| **악의적 P-TA** | 테넌트 관리자 자격으로 타 테넌트·상위 권한을 노림 | 테넌트 격리 우회(→다른 테넌트), 빌링 조작(→P-CSP) 시도 |
| **악의적/실수한 P-CSP** | 가격 정책 권한으로 빌링을 부당 조작 | SoD 우회로 자기 승인, 단가 소급 변경 시도 |
| **악의적/침해된 P-OP** | 최고 권한 운영자. 계정 침해 시 영향 최대 | 감사 로그 변조·시크릿 탈취 시도 (내부자 위협) |
| **침해된 노드/외부 시스템** | 침해된 Proxmox 노드, 위조된 IdP 응답, 악성 PG(향후) | TB-3·TB-4·TB-5 너머에서 신뢰 영역을 오염 |

> 권한 상승 방향은 **P-EU → P-TA → P-CSP → P-OP** ([`../00-overview/personas.md`](../00-overview/personas.md) §7). 위협 모델은 인접 페르소나 간 상승을 항상 가정한다. **P-CSP(가격 정책)와 P-OP(인프라 운영)는 SoD로 분리**되어, 1인 운영 환경이라도 동일 자격증명으로 합치지 않는다.

## 2. 보호 자산 (Assets)

| 자산 | 민감도 | 침해 시 영향 |
|---|---|---|
| 사용자 자격증명·세션 토큰 | 높음 | 계정 탈취 → 권한 상승 진입점 |
| 테넌트 자원·메타데이터 | 높음 | 테넌트 격리 실패 = 가치 명제 정면 부정 (R-010 치명적) |
| 미터링 원천 데이터 | 높음 | 변조 시 빌링 무결성 붕괴 (R-006 치명적) |
| 가격 정책·청구서 | 높음 | 부당 과금·감사 실패 |
| 감사 로그 | 높음 | 변조 시 부인 방지·감사 가능성 상실 |
| 시크릿 (Proxmox Token·JWT 키·OIDC·DB·PG 자격증명) | 최고 | 유출 시 전 영역 횡적 이동 (TB-7) |
| Proxmox 노드 제어 채널 | 높음 | 데이터 플레인 직접 조작 (R-004·R-009) |

## 3. 신뢰 경계별 STRIDE 위협

> 각 경계의 정의·통제는 [`../01-architecture/trust-boundaries.md`](../01-architecture/trust-boundaries.md) 참조. 아래는 그 위협 시드를 정식 위협으로 전개한 것이다.

### TB-1 — 인터넷 ↔ 컨트롤 플레인

| ID | STRIDE | 위협 | 대응 통제 (v0 방향) |
|---|---|---|---|
| TH-01 | S | 자격증명 탈취·세션 위조로 타 사용자 가장 | 강한 인증(JWT 서명 검증, Phase 2 OIDC/MFA), 짧은 세션 수명, TLS 전 구간 |
| TH-02 | T | 요청 본문·파라미터 변조로 인가 우회 | 서버측 입력 검증, 인가는 토큰 클레임이 아닌 서버측 정책으로 결정 |
| TH-03 | R | 사용자가 자기 행위를 부인 | 모든 상태 변경을 누가/언제/무엇/어디서 + 페르소나 코드로 감사 로그 기록 |
| TH-04 | I | 에러 메시지·응답에 내부 정보 노출 | 일반화된 에러 응답, 스택 트레이스·내부 식별자 비노출 |
| TH-05 | D | 무차별 요청으로 가용성 저하 | 레이트 리밋, 인증 실패 백오프 (Phase 1 최소, Phase 2+ 강화) |

### TB-2 — 테넌트 A ↔ 테넌트 B

| ID | STRIDE | 위협 | 대응 통제 (v0 방향) |
|---|---|---|---|
| TH-06 | I | 테넌트 필터 우회로 타 테넌트 자원·메타데이터 열람 | **tenant_id + PostgreSQL RLS + 앱 쿼리 필터 이중 방어, fail-closed(테넌트 컨텍스트 없으면 0행)** ([ADR-0005](../02-adr/0005-multitenancy-isolation-model.md)). R-010 |
| TH-07 | E | 권한 상승으로 타 테넌트 자원 조작 (P-TA→타 테넌트) | 인가 미들웨어가 RLS 세션 변수 주입 + RLS deny 기본 정책(ADR-0005), 우회 경로 발견 시 즉시 위협 모델 갱신. Phase 2 악의적 테넌트 회귀 테스트(CI 게이트) |
| TH-08 | T | 공유 식별자(예: 예측 가능한 자원 ID) 추측으로 교차 접근 | 비순차 식별자, 객체 수준 인가(소유 테넌트 확인) |

> TB-2는 R-010(멀티테넌시 격리 실패, **치명적**)의 위협 모델 측 표현이다. [ADR-0005](../02-adr/0005-multitenancy-isolation-model.md)(Accepted)가 4층 격리(메타데이터=tenant_id+RLS+앱필터 이중 방어, 스토리지=메타귀속+쿼터, 네트워크=Phase 5 이연, 컴퓨팅=Phase 5 사이드채널)를 확정했다. 네트워크·컴퓨팅 강한 격리는 Phase 5로 이연되므로, 그 전까지 다중 테넌트 네트워크 격리는 **약한 상태**임을 한계로 명시한다(CSAP 매핑에도 반영 예정).

### TB-6 — 일반 운영 영역 ↔ 빌링·감사 고신뢰 영역 (SoD)

| ID | STRIDE | 위협 | 대응 통제 (v0 방향) |
|---|---|---|---|
| TH-16 | T | 미터링 원천 데이터 변조 | 미터링은 **append-only**(DB 권한 미부여 + 트리거 이중 차단), 정정은 보정 레코드(`corrects_id`)로만. **레코드 해시 체인**으로 변조·삭제·순서 변경 탐지 ([ADR-0006](../02-adr/0006-billing-integrity-mechanism.md)). R-006 |
| TH-17 | T | 단가 테이블 직접 UPDATE·소급 변경 | 단가 직접 UPDATE 금지, 신규 레코드+유효기간 방식, 변경 이력 테이블 |
| TH-18 | R | 단가 변경·수동 조정 행위 부인 | 빌링 감사 로그를 **운영 로그와 별도 스키마+권한**에 기록(ADR-0006), 변경 주체·시점 기록, 감사 로그도 해시 체인 |
| TH-19 | E | SoD 우회로 자기 요청을 자기 승인 (P-CSP) | 가격 정책 write와 빌링 발행 권한 분리, 충돌 권한 동일 자격증명 금지 (personas §7) |
| TH-20 | T | 청구서 비재현(idempotency 결여)으로 금액 불일치 | 같은 (테넌트, 기간, 미터링 집합, 단가 버전) 입력 → 같은 청구서 해시(머클 루트). idempotent 생성, 재발행은 동일 해시 재확인 ([ADR-0006](../02-adr/0006-billing-integrity-mechanism.md)·ADR-0008) |

> TB-6은 R-006(빌링 무결성, **치명적**)의 위협 모델 측 표현이다. [ADR-0006](../02-adr/0006-billing-integrity-mechanism.md)(Accepted)이 무결성 자료구조를 **해시 체인(개별 미터링 레코드 — 순서·변조 탐지) + 머클 루트(기간 청구서 — 외부 감사자의 부분 검증)** 조합으로 확정했다. append-only는 DB 권한 미부여 + 트리거 이중 차단으로 물리 강제하고, 정정은 보정 레코드로만 한다. 빌링 감사 로그는 별도 스키마+권한에 분리 보관한다.

### TB-7 — 애플리케이션 ↔ 시크릿 저장소

| ID | STRIDE | 위협 | 대응 통제 (v0 방향) |
|---|---|---|---|

> TB-7 소비자는 **어댑터·API·빌링** 3개다 ([`../01-architecture/c4-level2-containers.md`](../01-architecture/c4-level2-containers.md), trust-boundaries TB-7). 시크릿 범위는 Proxmox API Token, DB 자격증명, JWT 서명 키, OIDC client secret, PG 어댑터 자격증명을 포함한다.
