# ADR-0007: 시크릿 관리 방식

- **상태(Status)**: Proposed
- **결정일(Date)**: TBD (Phase 1 진입 전 확정)
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0002 (Proxmox API Token 사용)

## Context — 맥락

본 플랫폼은 다음 시크릿을 다룬다.

- **Proxmox API Token**: 노드당 1개 이상. 로그 유출 절대 금지 (ADR-0002).
- **DB 자격증명**: 컨트롤 플레인 ↔ DB.
- **JWT 서명 키 + JWK 회전**: API 인증 (Phase 1) + OIDC (Phase 2).
- **시크릿 암호화 마스터 키**: SOPS/Vault 등.
- **PG 어댑터 자격증명**: Mock 단계에서도 인터페이스 일치 필요 (Phase 3).
- **외부 IdP 클라이언트 시크릿**: OIDC (Phase 2).
- **이메일/알림 전송 자격증명**: 예산 알림 등.

내부 에이전트 지침의 보안 규칙은 "시크릿은 코드·로그·문서에 절대 평문으로 남기지 않는다"고 못박았고, 그 구체 방식을 본 ADR에 위임했다. 본 ADR은 Phase 1 진입 전 확정해야 한다 (Phase 1 코드에 시크릿 첫 도입).

본 결정은 다음 두 축으로 갈린다.

- **저장**: 시크릿이 어디에 어떤 형태로 저장되는가? (파일 + 마스터 키 / 외부 시크릿 매니저 / 둘 다)
- **회전·만료·감사**: 시크릿 회전 절차, 만료 정책, 누가 언제 시크릿을 봤는지 감사 로그.

## Decision — 결정

[Phase 1 진입 전 확정.]

본 ADR이 다룰 결정 항목:

- Phase 1 시점의 시크릿 저장 방식: **SOPS + age/GPG / Vault dev 모드 / 또 다른 옵션**.
- 시크릿이 런타임 메모리에 어떻게 들어오는가 (env / 파일 마운트 / 시크릿 매니저 API).
- 로그·에러 메시지·예외 트레이스에 시크릿이 절대 포함되지 않게 강제하는 코드 레벨 메커니즘 (구조화 로깅의 redact 정책 등).
- Phase 2 OIDC 도입 시 시크릿 관리 모델의 확장 경로.
- 시크릿 접근에 대한 감사 로그 (누가·언제·어떤 시크릿을 조회했는가).
