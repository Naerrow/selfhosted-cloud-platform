# ADR-0007: 시크릿 관리 방식

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-06-23
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0002 (Proxmox API Token 사용), 0004 (Go + 단일 바이너리 + 단일 PostgreSQL), 0009 (내부 전용 산출물), 0011 (Apache-2.0 라이선스)

## Context — 맥락

본 플랫폼은 다음 시크릿을 다룬다.

- **Proxmox API Token**: 노드당 1개 이상. 로그 유출 절대 금지 (ADR-0002).
- **DB 자격증명**: 컨트롤 플레인 ↔ DB.
- **JWT 서명 키 + JWK 회전**: API 인증 (Phase 1) + OIDC (Phase 2).
- **시크릿 암호화 마스터 키**: 본 ADR에서 확정하는 저장소의 루트 키.
- **PG 어댑터 자격증명**: Mock 단계에서도 인터페이스 일치 필요 (Phase 3).
- **외부 IdP 클라이언트 시크릿**: OIDC (Phase 2).
- **이메일/알림 전송 자격증명**: 예산 알림 등.

내부 에이전트 지침의 보안 규칙은 "시크릿은 코드·로그·문서에 절대 평문으로 남기지 않는다"고 못박았고, 그 구체 방식을 본 ADR에 위임했다. 본 ADR은 Phase 1 진입 전 확정해야 한다 (Phase 1 코드에 시크릿 첫 도입).

위협 모델(STRIDE v0)의 **TB-7**이 직접 입력이다 — TH-21(시크릿이 로그·에러·트레이스에 평문 노출), TH-22(한 소비자의 redact 누락이 전 시크릿 노출로 확대), TH-23(탈취된 JWT 서명 키로 임의 토큰 위조). 추가로 TB-3의 TH-09(Proxmox API Token 유출 시 노드 직접 조작·회전)도 본 ADR의 조달·회전 결정에 연결된다.

본 결정은 다음 두 축으로 갈린다.

- **저장**: 시크릿이 어디에 어떤 형태로 저장되는가? (파일 + 마스터 키 / 외부 시크릿 매니저 / 둘 다)
- **회전·만료·감사**: 시크릿 회전 절차, 만료 정책, 누가 언제 시크릿을 봤는지 감사 로그.

## Decision — 결정

시크릿 관리 방식을 다음과 같이 확정한다. ADR-0004(단일 바이너리·단일 PostgreSQL·1인 운영)·vision §2(셀프호스팅·데이터 주권)를 전제로 한다.

1. **저장 — SOPS + age**:
   - 시크릿은 **SOPS로 암호화된 파일**로 보관하고, 복호화 키는 **age**(X25519 기반) 키쌍으로 둔다. 값은 AES-256-GCM으로 암호화되어 평문이 아니다. age **개인 키(마스터 키)는 git 밖**에 둔다 — 배포 호스트의 파일 권한 제한 경로(예: `0600`) 또는 환경 주입.
   - **공개/내부 경계 (ADR-0009 정합)**: 실제 암호화 시크릿 파일(예: `secrets.enc.yaml`)은 **공개 리포에 두지 않는다** — `.gitignore`로 추적 제외하고 ADR-0009의 내부 전용 산출물로 다룬다. 공개 리포에는 **키 이름만 있고 값이 빈 템플릿**(예: `secrets.example.yaml`)만 둔다. 암호문조차 공개하지 않으므로 오프라인 크랙·장기 양자내성 노출을 차단한다(암호화 파일 git 버전 관리 이점은 의도적으로 포기). **SOPS 설정 파일(`.sops.yaml`) 자체**는 키 ID·암호화 규칙만 담아 평문 시크릿이 없으므로 공개 가능하다.
   - 외부 시크릿 매니저(HashiCorp Vault 등)는 **도입하지 않는다**. 별도 프로세스·네트워크 의존성이 ADR-0004의 단일 바이너리 결정과 1인 운영 부담에 어긋나고, 클라우드 KMS는 vision §2의 데이터 주권 가치와 충돌한다(대안 비교 참조).
   - 평문 시크릿은 어떤 경우에도 git·로그·문서에 남기지 않는다.
2. **런타임 조달 — 시작 시 복호화 → 메모리**:
   - **복호화 책임은 애플리케이션 프로세스 내부**에 둔다 — Go SOPS 디코딩 라이브러리(import path `github.com/getsops/sops/v3/decrypt` — SOPS가 안정성을 보장하는 유일한 외부 API; `decrypt.File(path, format)` / `decrypt.Data(data, format)`)[^sops-decrypt]를 바이너리에 임베드해, 기동 시 암호화 파일을 **메모리에서 직접 복호화**한다. 별도 `sops` CLI를 셸로 호출하거나 평문을 파일/`tmpfs`에 풀어 entrypoint가 주입하는 방식은 쓰지 않는다(평문 파일 잔류·프로세스 인자 노출 회피). 복호화된 시크릿은 **프로세스 메모리에만** 상주한다.
   - 시크릿은 **타입드 설정 구조체**로만 전달하고, 전역 환경변수에 평문을 흩뿌리지 않는다 — env 경유 노출(프로세스 목록·`/proc`·크래시 덤프)을 최소화한다.
   - 시크릿 **소비자는 어댑터·API·빌링 3개**다(위협 모델 TB-7). 단, 본 구조는 **단일 프로세스 모듈러 모놀리스**(ADR-0004)이므로 소비자 분리는 OS·프로세스 수준의 강한 격리가 아니라 **코드 구조 규율**이다 — 각 소비자는 **자기 종류의 시크릿만 받는 provider 인터페이스**로 접근하고(Proxmox Token / JWT 키 / OIDC / DB / PG 분리), 전역에서 임의 시크릿을 끌어 쓰지 못하게 한다. 이로써 한 소비자의 redact 누락이 전 시크릿 노출로 번질 표면을 줄인다(TH-22 부분 완화 — 강한 격리는 모듈→서비스 분리 시점인 Phase 4+ 재검토).
3. **로그 노출 차단 — 코드 레벨 강제 (TH-21)**:
   - 시크릿을 담는 타입은 `String()`/`MarshalJSON`/로그 출력에서 **항상 redact**되도록 정의한다(예: 마스킹 래퍼 타입). 구조화 로깅은 **allowlist 기반 필드 출력**을 기본으로 하여, 시크릿 필드가 우발적으로 직렬화되지 않게 한다.
   - 에러·예외·스택 트레이스에 시크릿 값이 실리지 않도록, 시크릿을 에러 메시지에 포매팅하는 것을 금지한다(코드 리뷰·linter로 강제).
4. **회전·만료**:
   - **Proxmox API Token**은 토큰별 권한 분리(privilege separation)와 **만료일**을 부여한다(사전 조사 자료 §3.3 [P1]/[P3] — `pveum` 토큰은 사용자별 발급·별도 권한·만료일 지원, 유출 시 사용자 비활성화 없이 token만 폐기). 유출 의심 시 **사용자가 아니라 token만 폐기·재발급**한다.
   - **JWT 서명 키**는 회전을 전제로 설계한다 — Phase 1은 단일 활성 키 + 폐기 절차, Phase 2 OIDC에서 **JWK 회전(kid 기반 다중 키)** 으로 확장한다(키 ID로 검증 키를 선택). 탈취 시 즉시 폐기·재발급(TH-23 완화).
   - Phase 1 회전은 **수동(런북 기반)** 으로 한다. 자동 회전은 시크릿 종류·운영 부담을 보고 후속 Phase에서 재검토한다(본 ADR의 한 한계 — Consequences 참조).
5. **시크릿 유출 대응 런북 (Phase 1 최소 1건)**:
   - 시크릿 유출 의심 시 **폐기·재발급 절차**를 런북으로 둔다(`docs/04-operations/runbooks/`). 최소 Proxmox API Token과 JWT 서명 키 2종을 대상으로 Phase 1에 1건 이상 작성한다(위협 모델 TH-23 연결).
6. **접근 감사의 한계 명시**:
   - SOPS+age는 **시크릿 접근 자체의 세밀한 감사 로그를 제공하지 않는다**(파일 복호화 단위). "누가·언제·어떤 시크릿을 조회했는가"의 통제는 본 모델에서 **약하며**, 이를 v0의 명시적 한계로 둔다. 강한 접근 감사가 요구되면 Vault류 도입을 신규 ADR로 재평가한다(회귀 트리거).

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **SOPS + age** | **외부 런타임 서비스·네트워크 의존 없음**(복호화 라이브러리는 바이너리에 임베드 — 별도 프로세스/데몬 불필요, 단일 바이너리 정합), 셀프호스팅·데이터 주권 가치 정합, 미니 PC 운영 부담 최소 | 시크릿 회전 수동, 접근 단위 감사 약함(파일 복호화 단위), 라이브러리 의존성은 존재(SBOM 추적) | **채택** |
| HashiCorp Vault (dev 모드) | 동적 시크릿·lease·회전·접근 감사 device 표준 제공 | 별도 프로세스·네트워크 의존성(ADR-0004 단일 바이너리와 마찰), 1인 운영 부담↑, dev→HA 이전 비용 | 거부 — 운영 부담이 1인 가능 범위를 압박. 강한 회전·감사 요구 시 회귀 트리거로 재평가 |
| Vault dev → Vault HA (Phase 4+) | 점진 도입 경로 | 초기부터 의존성·이전 비용 선지불 | 거부 — 본 단계에서 과함. Phase 5 시크릿 폭증 시 재검토 항목으로만 유지 |
| 클라우드 KMS (AWS KMS 등) | 마스터 키 운영 부담↓ | 외부 클라우드 의존 → vision §2 데이터 주권 가치와 정면 충돌 | 거부 |
| OS 키링 | 단일 노드 단순 | 멀티 노드(Phase 4+) 환경 부적합, git 흐름과 불일치 | 거부 |
| env 평문 + `.gitignore` | 가장 단순 | 평문 시크릿이 디스크·프로세스 환경에 상주, 백업·덤프 유출 위험 | 거부 — "평문 금지" 원칙 정면 위반 |

## Consequences — 결과

### 긍정적
- **외부 런타임 서비스 없이** 단일 바이너리에 복호화 라이브러리를 임베드해 시크릿을 다룬다(별도 프로세스/데몬·네트워크 의존 없음) — ADR-0004(단일 바이너리·1인 운영)·vision §2(셀프호스팅·데이터 주권)와 정합한다. 복호화 라이브러리 의존성 자체는 SBOM·라이선스 스캔(R-008)으로 추적한다.
- 시크릿 저장 형식·SOPS 규칙(`.sops.yaml`)·키 이름 템플릿이 코드와 함께 관리되어 **구성 의도가 리포에 드러난다**(실제 암호문·평문은 공개하지 않음 — 내부 전용).
- redact 래퍼 타입 + 구조화 로깅 allowlist로 위협 모델 **TH-21**을 코드 레벨에서 강제하고, provider 인터페이스로 소비자 접근 표면을 좁혀 **TH-22**를 부분 완화한다(단일 프로세스이므로 강한 격리가 아닌 코드 규율임은 한계로 명시).
- Proxmox Token 만료일·token-only 폐기, JWT 키 회전(Phase 2 JWK)로 **TH-09·TH-23**의 회전·폐기 경로를 확보한다.

### 부정적
- **시크릿 회전이 수동(런북 기반)** 이다 — 회전 누락 위험이 운영자에게 남는다. 회전 주기·자동화는 후속 Phase 재검토 과제로 둔다.
- **접근 단위 감사가 약하다**(파일 복호화 단위, 시크릿별 조회 로그 없음). 내부자(P-OP)의 시크릿 조회를 세밀히 감사할 수 없다 — 이를 명시적 한계로 두고, 강한 감사 요구 시 Vault류 신규 ADR 트리거.
- age 마스터 키 자체의 보관·백업·분실 복구가 **운영자 단일 책임**으로 남는다 — 마스터 키 분실 시 전 시크릿 복호화 불가. 키 백업 절차를 런북에 포함해야 한다.

### 중립적
- 본 결정은 위협 모델 **TB-7(TH-21·22·23)** 의 통제를 "확정 예정" → 구체 통제로 확정한다. 위협 모델 v0의 "ADR-0007 Accepted 시 TB-7 통제 구체화·시크릿 유출 런북 연결" 트리거가 본 ADR로 충족된다. TB-3의 TH-09(Proxmox Token 회전)도 본 ADR의 만료·폐기 결정으로 보강된다.
- SOPS·age 모두 본 프로젝트 라이선스 Apache-2.0(ADR-0011)과 호환된다(SOPS = MPL-2.0, age = BSD-3-Clause — 둘 다 Apache-2.0 배포와 정합). 외부 의존성 라이선스는 Phase 1 CI의 SBOM·라이선스 스캔으로 지속 검증(R-008).
- 시크릿 관리가 새 외부 도구(SOPS·age)를 도입하므로 리스크 레지스터에 **R-011(시크릿 유출·관리)** 을 신설한다(risks.md 회귀 트리거 "새 외부 의존성 도입").

## Open Questions

> 핵심 결정(저장 = SOPS+age, 런타임 = 시작 시 복호화→메모리, 로그 redact 강제, Proxmox Token 만료·JWT 키 회전, 유출 런북)은 위에서 확정됐다. 남은 항목은 구현 세부와 후속 트리거다.

- 시크릿 회전 주기와 자동화 수준 — Phase 1은 수동(런북)으로 확정. 반자동·자동화는 Phase 2(OIDC 도입으로 키 종류 증가) 시점에 재검토.
- age 마스터 키의 **백업·분실 복구 절차**(복수 수신자 키 / 오프라인 백업) — Phase 1 시크릿 유출 런북과 함께 구체화.
- 구조화 로깅의 redact 구현 — allowlist 직렬화 vs 마스킹 래퍼 타입의 최종 조합은 Phase 1 로깅 설계 시 확정(결정 영향 작음).
- 시크릿 파일 배치는 본 ADR Decision에서 확정했다 — **실제 암호화 파일(`secrets.enc.yaml`)은 `.gitignore`로 추적 제외(ADR-0009 내부 전용), 공개 리포에는 빈 값 템플릿(`secrets.example.yaml`)과 평문 없는 SOPS 설정(`.sops.yaml`)만 둔다.** `.gitignore` 패턴(`secrets.enc.yaml`·`*.enc.yaml`·`*.enc.json`·`*.enc.env`)은 본 결정과 함께 추가했다(example·`.sops.yaml`은 추적 유지 확인). 템플릿·디렉터리 경로의 구체 정착은 Phase 1 리포 구조 확정 시 마무리(결정 영향 작음).
- **회귀 트리거 (강한 접근 감사)**: 외부 감사·내부자 위협 통제로 "누가·언제 시크릿을 조회했는가"의 세밀한 감사가 요구되면, 거부했던 Vault(audit device)를 신규 ADR로 재평가한다.
- **회귀 트리거 (Phase 5 시크릿 폭증)**: Adjacent Services 도입으로 시크릿 종류가 폭증하면 SOPS 파일 관리·회전 부담이 본 모델의 한계를 넘는지 재평가한다.
