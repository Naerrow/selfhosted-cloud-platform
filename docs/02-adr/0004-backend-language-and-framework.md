# ADR-0004: 백엔드 언어/프레임워크 선택

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-06-06
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0002 (Proxmox 채택), 0003 (자체 오케스트레이션), 0005 (멀티테넌시 격리), 0006 (빌링 무결성), 0011 (Apache-2.0 라이선스)

## Context — 맥락

본 플랫폼의 컨트롤 플레인은 Proxmox REST API를 어댑터로 격리한 위에(ADR-0002·0003) 다음 책임을 가진다.

- HTTP/REST API + WebSocket(noVNC 프록시 등)
- 노드·VM 메타데이터 영속화 (관계형 DB)
- 미터링 파이프라인 (수집 → 정규화 → 시계열 저장)
- 백그라운드 작업 (스케줄러·페일오버 감지·청구서 생성)
- 멀티테넌시 강제 (모든 쿼리에 테넌트 ID 필터)
- 빌링 무결성 (append-only, 해시 체인 + 기간 머클 루트 — ADR-0006)

언어·프레임워크 선택은 다음에 직접 영향을 준다.

- **개발 속도 vs 런타임 성능**: Phase 1 시연 일정과 Phase 4+ 다중 노드 부하 사이 균형.
- **타입 안전성**: 멀티테넌시 격리·빌링 무결성을 컴파일 타임에 강제할 수 있는가.
- **에코시스템**: Proxmox API 클라이언트 라이브러리, OIDC, 시계열 DB 클라이언트, PDF 생성기.
- **운영 부담**: 1인 운영 가능 범위 안에서 런타임·의존성 관리가 단순한가.
- **라이선스 호환성**: AGPLv3 Proxmox 의존성과 본 프로젝트 라이선스(미정, ADR-별도) 정합.

본 ADR은 Phase 1 진입 전 확정해야 한다. 결정이 지연되면 Phase 1 코드가 즉시 막힌다.

## Decision — 결정

백엔드 스택과 구조를 다음과 같이 확정한다.

1. **언어/프레임워크**: **Go + Echo**(HTTP) + **sqlc**(SQL→타입 생성). Echo는 Gin과 동급의 성숙한 라우터이며, WebSocket(noVNC 프록시)·미들웨어 생태계가 충분하다. sqlc는 손으로 쓴 SQL을 컴파일 타임에 검증된 타입 안전 코드로 변환해, ORM 추상화 없이 멀티테넌시 필터·빌링 쿼리를 명시적으로 다룬다.
2. **구조**: **모듈러 모놀리스**. 단일 바이너리·단일 배포로 시작하되, 내부를 `controlplane`(API·오케스트레이션·어댑터) / `metering` / `billing` 모듈 경계로 분리한다. 모듈 간 호출은 인터페이스를 통하며, 다른 모듈의 DB 테이블을 직접 건드리지 않는다. Phase 4+ 부하·운영 요구가 실증되면 모듈을 별도 서비스로 분리하는 경로를 열어둔다.
3. **데이터 저장소**: **PostgreSQL 단일 인스턴스**. 메타데이터, 미터링(시간 파티셔닝), 감사 로그(별도 스키마+권한)를 한 인스턴스에서 다루되 스키마·권한으로 경계를 둔다. 빌링 감사 로그는 별도 스키마로 분리해 운영 로그와 권한을 가른다(ADR-0006 무결성 메커니즘과 연결). 미터링 시계열이 단일 PostgreSQL 한계를 넘어서면 TimescaleDB 확장 또는 전용 시계열 DB 분리를 별도 ADR로 재검토한다.
4. **백그라운드 작업**: Phase 1은 **동기 처리 + 주기 실행(in-process ticker/cron)** 으로 충분히 다룬다. 본격 작업 큐(예: Asynq — Redis 기반 Go 큐)는 **Phase 3(빌링) 진입 시** 도입한다. 청구서 생성·미터링 집계의 비동기·재시도·idempotency 요구가 그 시점에 실질화되기 때문이다.
5. **프런트엔드**: 웹 콘솔은 별도 SPA로 두며 백엔드와 같은 언어로 묶지 않는다. 백엔드는 REST/JSON + WebSocket만 제공하고, 콘솔·CLI는 그 위의 클라이언트다([C4 Level 2](../01-architecture/c4-level2-containers.md)에서 클라이언트는 컨트롤 플레인 경계 밖).

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **Go + Echo + sqlc** | 단일 바이너리(미니 PC 배포·1인 운영에 최적), 저메모리, 강한 동시성(헬스체크·폴링·스케줄러), 정적 타입, Proxmox/OIDC 클라이언트 성숙, sqlc로 명시적 타입 안전 SQL | 제네릭 이전 패턴 일부 잔존, 보일러플레이트, ORM 편의 부재(의도된 trade-off) | **채택** |
| Python + FastAPI + SQLAlchemy | 생태계·생산성·문서화 우수, 빠른 시연 | 런타임 성능·GIL, 배포 복잡도(인터프리터+의존성), 멀티테넌시 강제가 런타임 의존 | 거부 — 1인 운영의 배포 단순성과 동시성에서 Go에 밀림 |
| TypeScript(Node) + NestJS + Prisma | 프런트와 단일 언어, Prisma 타입 | Node 메모리·운영 모델, CPU 바운드(해시 체인 검증 등) 약점, 백그라운드 안정성 | 거부 — 빌링 무결성의 CPU 작업·운영 안정성에서 불리 |
| Rust + Axum + SQLx | 메모리 안전·최고 성능, 컴파일 타임 보장 강력 | 개발 속도 비용 큼, 1인 파트타임 시연 일정과 충돌 | 거부 — 깊이 가치엔 부합하나 Phase 1 속도·1인 범위에서 과함 |
| Elixir + Phoenix | 동시성·내결함성 탁월 | 1인 운영자 학습 곡선, 생태계(Proxmox/시계열 클라이언트) 상대적 부족 | 거부 — 학습 비용이 1인 운영 가능 범위를 압박 |

## Consequences — 결과

### 긍정적
- **단일 바이너리 배포**가 미니 PC~서버 혼합 환경(vision §3.2)과 1인 운영에 정합한다. 런타임·의존성 관리 부담이 최소.
- **sqlc의 명시적 SQL**로 모든 자원 쿼리의 테넌트 필터를 코드 리뷰에서 눈으로 확인 가능 — 멀티테넌시 격리(ADR-0005)·빌링 무결성(ADR-0006)을 ORM 마법 없이 명시적으로 강제.
- Go의 경량 동시성이 헬스체크·하트비트·폴링 기반 미터링·스케줄러(ADR-0003)에 직접 들어맞는다.
- 모듈러 모놀리스는 **단일 배포의 단순함**과 **모듈 경계의 분리 가능성**을 동시에 확보 — Phase 4+ 분리 경로를 닫지 않는다.

### 부정적
- ORM 편의(자동 마이그레이션·관계 로딩)를 포기한다 → 마이그레이션 도구(예: golang-migrate)와 SQL을 직접 관리.
- 모듈 경계 규율을 사람이 지켜야 한다(다른 모듈 테이블 직접 접근 금지). 위반 시 모놀리스가 진흙공이 됨 → 코드 리뷰·linter로 경계 강제.
- 단일 PostgreSQL에 미터링 시계열을 얹으면 쓰기 부하 증가 시 병목 가능 → 파티셔닝으로 1차 대응, 한계 시 분리(회귀 트리거).

### 중립적
- Go(BSD 계열)·Echo(MIT)·sqlc(MIT) 모두 본 프로젝트 라이선스 Apache-2.0(ADR-0011)과 호환되고, AGPLv3는 Proxmox **API 클라이언트**로만 닿으므로 본 코드에 AGPL 전염이 발생하지 않는다(네트워크 경유 호출, 코드 링크 아님). 외부 의존성 라이선스는 Phase 1 CI의 SBOM·라이선스 스캔으로 지속 검증(R-008).

## Open Questions

> 본 ADR의 핵심 결정(언어·구조·DB·큐 시점)은 위에서 확정됐다. 남은 항목은 구현 시점 세부다.

- sqlc + golang-migrate 조합 vs 다른 마이그레이션 도구의 최종 선택 — Phase 1 구현 착수 시 확정(결정 영향 작음).
- 미터링 파티셔닝 전략(기간·테넌트 기준)의 구체 키 — Phase 3 미터링 파이프라인 설계 시.
- 작업 큐 도입 시 Asynq vs 대안(River 등 신규 Go 큐)의 최종 비교 — Phase 3 진입 시점 재평가.
- **회귀 트리거**: Phase 5(Adjacent Services) 도입 시 코드 표면적 폭증을 모듈러 모놀리스가 감당 가능한가? 모듈→서비스 분리 필요성을 그 시점에 재평가.
