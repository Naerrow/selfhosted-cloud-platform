# ADR-0003: Proxmox 자체 클러스터링을 사용하지 않고 자체 오케스트레이션 레이어를 구현한다

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-05-25
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0002 (Proxmox 채택)

## Context — 맥락

Proxmox VE는 **Corosync** 기반 자체 클러스터링과 **pmxcfs**(분산 설정 파일시스템)를 내장한다. 한 클러스터에 묶인 노드들은 `/etc/pve`를 공유하고, HA Manager로 노드 장애 시 VM 자동 페일오버가 가능하다.

그러나 본 프로젝트의 비전은 다음을 요구한다.

1. **혼합 하드웨어 풀**: 서버 + 미니 PC + 노트북을 단일 콘솔로 운영
2. **지리적 분산 가능성**: 사용자의 자택·사무실·외부 데이터센터에 분산된 노드를 묶음
3. **강한 멀티 테넌시**: 여러 고객(테넌트)이 동일 풀을 공유하면서 강하게 격리되어야 함
4. **수평 확장 가능성**: 노드 수가 수십 대를 넘을 가능성을 열어둠

사전 조사 자료(`docs/06-research/cloud-platforms-and-regulations.md`)에 정리된 Proxmox 자체 클러스터링의 **알려진 한계**는 본 프로젝트의 요구와 정면으로 충돌한다.

| 한계 | 근거(요약) |
|---|---|
| 단일 데이터센터·저지연 LAN 전제 | 공식 매뉴얼은 명시적 노드 수 한계를 두지 않으나(50+ 노드 사례 보고), 실용적 운용은 단일 DC를 전제로 함 |
| 노드 간 지연 5ms 미만 요구 | 공식 매뉴얼 *"reliable network with latencies under 5 milliseconds (LAN performance) between all nodes"* — 광역(WAN)·다중 데이터센터 묶음에 부적합 |
| 단일 클러스터 = 단일 데이터센터 모델 | 멀티사이트는 별도 클러스터로 분리 필요 |
| Quorum 의존 (과반수 미달 시 쓰기 불가) | 분산 환경의 네트워크 분리에 취약 |
| 본격적 멀티 테넌시 부족 | 풀(Pool)·권한 단위 분리는 가능하나 네트워크·스토리지의 강한 격리는 추가 설계 필요 |
| 글로벌 스케줄러 부재 | 자원 여유 기반 자동 배치를 자체 구현하기 어려움 |

> 한계 항목은 작성일 기준이며, Proxmox의 향후 개선으로 변할 수 있다. 그러나 본 프로젝트의 시점에서 이 제약은 명백하다.

## Decision — 결정

**본 플랫폼은 Proxmox 자체 클러스터링(Corosync + pmxcfs + HA Manager)을 사용하지 않는다.**

대신, 각 Proxmox 노드는 **독립 노드**로 운영되며, 본 플랫폼의 **자체 오케스트레이션 레이어**가 컨트롤 플레인에서 다음을 책임진다.

1. **노드 등록/해제**: API Token으로 독립 Proxmox 노드를 등록.
2. **헬스 체크·하트비트**: 노드 가용성 추적.
3. **자원 풀링**: 여러 독립 노드를 하나의 논리적 풀로 추상화.
4. **스케줄링**: VM/CT 생성 시 자원 여유·태그·정책 기반 노드 선택.
5. **장애 감지**: 노드 다운 시 알림 및 (추후) 다른 노드로 워크로드 재배치.
6. **마이그레이션**: 마이그레이션(특히 라이브 마이그레이션)은 Phase 4 이후 별도 ADR에서 공유 스토리지·이미지 전송·네트워크 전제를 포함해 결정한다. 독립 노드 모델에서는 단순 API 위임만으로 성립하지 않기 때문이다. 필요 시 실제 실행은 Proxmox API를 사용할 수 있으나, **정책 결정과 가능 여부 판단의 주체는 본 플랫폼**이다.

즉, Proxmox는 **단일 노드 하이퍼바이저**로만 사용하고, 노드 간 협조는 본 플랫폼이 책임진다.

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **Proxmox 자체 클러스터링 사용** | 즉시 사용 가능, HA Manager 내장, pmxcfs로 설정 일관성 보장 | 위 한계 6종 모두 그대로 적용, 본 프로젝트의 비전과 정면 충돌 | 거부 |
| **Proxmox 클러스터 + 본 플랫폼 혼합** | 작은 묶음(같은 사이트)은 Proxmox 클러스터로, 묶음 간은 본 플랫폼으로 | 두 계층의 책임 경계 모호, 장애 모드 폭증, 운영자 인지부하 큼 | 거부 |
| **각 노드 독립 + 본 플랫폼 단독 오케스트레이션** | 모든 책임이 본 플랫폼 한 곳에 응집, 비전과 정렬, 멀티사이트 자연 지원 | 본 플랫폼이 직접 클러스터링·스케줄링·페일오버를 구현해야 함 (개발 공수 큼) | **채택** |
| **Kubernetes + KubeVirt** | K8s 생태계 활용, VM도 컨테이너처럼 다룸 | K8s 자체가 미니 PC 환경에 무겁고, "Proxmox 기반 셀프호스팅 클라우드"라는 프로젝트 정체성과 어긋남 | 거부 |

## Consequences — 결과

### 긍정적
- **단일 LAN·quorum 전제에서 자유로움**: Proxmox 자체 클러스터링의 저지연 LAN·quorum 요구가 본 플랫폼의 노드 관리에는 적용되지 않는다.
- **지리적 분산 가능**: 노드들이 같은 LAN에 있을 필요가 없다(VPN/공개 인터넷 + API만 가능하면 됨).
- **멀티 테넌시를 본 플랫폼이 일관되게 강제**: Proxmox 풀/권한이 아니라 본 플랫폼의 RBAC + 네트워크/스토리지 격리로 처리.
- **글로벌 스케줄러를 본 플랫폼이 가짐**: 자원 배치·이전 결정의 주체가 명확.
- **본 프로젝트의 핵심 가치(컨트롤 플레인 설계)** 가 자연스럽게 부각된다. 이는 본 프로젝트가 "Proxmox에 무엇을 더하는가"라는 정체성을 분명히 한다.

### 부정적
- **HA Manager의 자동 페일오버를 잃는다**: 본 플랫폼이 이를 직접 구현해야 한다. Phase 4 이전까지는 **알림만** 제공하고, 자동 페일오버는 그 이후로 미룬다.
- **pmxcfs의 설정 일관성을 잃는다**: 설정 동기화를 본 플랫폼이 책임져야 한다. → 자체 메타데이터 DB가 단일 진실 공급원이 된다.
- **라이브 마이그레이션의 어려움**: 두 노드가 동일 클러스터에 있지 않으면 vMotion 유사 기능은 별도 설계가 필요하다(공유 스토리지, 이미지 전송 등). Phase 4에서 별도 ADR로 다룬다.

### 중립적
- 각 Proxmox 노드는 **독립 라이센스/업데이트 주기**를 따른다. 본 플랫폼은 노드별 Proxmox 버전 차이를 감지·호환해야 한다.
- 단일 노드만 등록한 환경에서도 본 플랫폼은 정상 동작해야 한다 (Phase 1의 가정).

## Operational Implications — 운영 영향

본 결정은 운영 모델에 다음 요구사항을 만든다.

- **노드 식별·태깅**: 노드별 사이트/존/하드웨어 클래스를 태그로 관리.
- **공유 스토리지 전략**: 라이브 마이그레이션·이미지 분산을 위해 어떤 스토리지를 공유할지 별도 ADR(Phase 5에서).
- **네트워크 모델**: 테넌트 격리는 본 플랫폼이 책임지므로, 노드 간 오버레이 네트워크(VXLAN 등) 설계가 필수 → Phase 5 네트워킹 ADR에서.
- **장애 모드 문서화**: 컨트롤 플레인 다운 시 데이터 플레인 영향(=기존 VM은 계속 동작, 신규 작업만 불가)을 런북에 명시.

## Open Questions

- 자동 페일오버(Phase 4 이후)의 정책은 어떻게 정의할 것인가? → 별도 ADR.
- 노드 간 메타데이터 동기화는 어떤 일관성 모델을 가질 것인가(strong vs eventual)? → 백엔드 아키텍처 ADR(0004 이후).
- 동일 데이터센터의 노드끼리만이라도 Proxmox HA를 부분적으로 활용할 가능성? → 일관성 우선, 현 시점 거부. 필요 시 별도 ADR로 재논의.
