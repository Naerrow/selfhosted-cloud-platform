# ADR-0008: Phase 3 빌링 스코프를 종량제 단일 모델로 한정한다

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-05-25
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0006 (빌링 데이터 무결성, Accepted)

## Context — 맥락

원 명세는 빌링 영역에 **다섯 가지의 별도 주제**를 함께 요구한다.

1. 자원 미터링
2. 가격 정책 관리
3. **세 가지 빌링 모델**: 종량제 / 약정형 인스턴스 / 구독형 정액 과금
4. 청구서 생성, PG 연동
5. **크레딧·할인·쿠폰**

또한 예산 알림, 비용 분석도 포함된다.

본 프로젝트의 가치 명제([vision.md](../00-overview/vision.md))는 **기능의 폭이 아니라 운영·보안·감사 가능성의 깊이**다. 빌링 영역에 이 명제를 적용하면, 모델 다양성(기능 폭)보다 **무결성·감사·권한 분리(SoD)** 의 깊이에 자원을 집중해야 한다. 모든 빌링 요구사항을 동일 비중으로 구현하면 핵심 가치가 희석된다.

## Decision — 결정

Phase 3의 빌링 스코프를 다음과 같이 한정한다.

### IN-SCOPE

1. **미터링 파이프라인** (수집 → 정규화 → 시계열 저장)
2. **가격 정책 엔진** (인스턴스 타입·스토리지·트래픽 단가)
3. **종량제(Pay-as-you-go) 단일 모델**
4. **청구서(Invoice) PDF 생성기**
5. **청구서·미터링 무결성 보장** (해시 체인 + 기간 머클 루트 — ADR-0006에서 확정)
6. **빌링 감사 로그** (단가 변경·수동 조정·청구서 재발행을 모두 추적)
7. **실시간 비용 대시보드**
8. **예산 알림**
9. **PG 어댑터 인터페이스 + Mock 구현체**

### OUT-OF-SCOPE (제외, 사유 명시)

| 제외 항목 | 사유 |
|---|---|
| **약정형 인스턴스** (Reserved) | 약정·해지·환불·만료 로직으로 빌링 엔진 복잡도가 2배 이상이 된다. 본 프로젝트의 깊이 항목(무결성·감사·SoD)에 기여하지 않으며, 빌링 엔진 표면적만 키운다. |
| **구독형 정액 과금** (Subscription) | 종량제와 청구 사이클·산정 방식이 완전히 다른 두 번째 모델이다. 한 모델을 깊게 파는 것이 더 가치 있다. |
| **크레딧 시스템** | PG 어댑터 + Mock 구현체 + 청구서 발행으로 결제 흐름은 충분히 시연 가능하다. 한국에서 실제 PG 연동(KCP/INICIS/Toss 등)은 개인사업자 등록·심사가 필요해 데모 환경에 부적합하다. 결과적으로 크레딧 자체가 불필요해진다. |
| **쿠폰/할인** | 순수 마케팅 기능이며 보안·운영·감사 관점의 함의가 거의 없다. 본 프로젝트의 가치 명제와 무관하다. |

### 핵심 깊이 요구사항 (남긴 항목에 대한 깊이 강화)

빌링 단순화로 절약된 공수는 다음 다섯 가지 **운영·보안 깊이**에 재투자한다.

1. **append-only 미터링**: 미터링 원천 데이터는 수정·삭제 불가. 정정은 보정 레코드(correction record)로만.
2. **단가 변경 추적**: 모든 가격 정책 변경은 변경자·시각·전후값을 감사 로그에 기록. 변경 자체에 별도 권한 필요.
3. **2-eye 승인 옵션**: 일정 임계치 이상의 단가 변경 또는 청구서 수동 조정은 두 번째 승인자 없이 적용되지 않는 흐름을 옵션으로 지원.
4. **Idempotent 청구서**: 같은 (테넌트, 기간) 입력에 대해 같은 청구서가 재생산된다. 청구서 재발행은 새 발행본을 만들지 않고 동일 해시를 재확인한다.
5. **빌링 감사 로그 분리 보관**: 일반 운영 로그와 다른 보관소(append-only 스토리지)에 저장하고, 무결성 검증 가능.

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **원 명세 전 항목 구현** (3개 모델 + 크레딧 + 쿠폰) | 명세 충실 | 핵심 가치 희석, 일정 폭증, 어느 것도 깊게 못 만들 위험 | 거부 |
| **종량제 단일 모델 + 깊이 강화** | 가치 명제와 정합, 일정 현실적, 무결성·SoD를 본격적으로 다룰 수 있음 | 명세 일부 미구현 | **채택** |
| **종량제 + 구독형 정액 과금 2개 모델** | "다중 모델 지원" 외형 확보 | 구독형 정액 과금의 추가 가치가 본 프로젝트 가치 명제(`vision.md`)에 맞지 않음 | 거부 |
| **모델은 종량제 하나, 그러나 크레딧은 유지** | "선불 결제 가능"이라는 흐름 보존 | 한국 환경에서 실제 PG 실연동 불가 → 크레딧 충전 자체가 Mock이 됨 → 무의미 | 거부 |

## Consequences — 결과

### 긍정적
- Phase 3에 빌링 **무결성·감사·SoD**를 깊게 다룰 시간이 확보된다.
- 빌링이 결제 기능이 아니라 **감사 대상 시스템**으로 다뤄지고 있다는 의도가 코드·문서·테스트에 일관되게 드러난다.
- 청구서 idempotency, 단가 변경 감사, append-only 미터링 같은 **재현 가능한 시연**이 가능해진다.
- 빌링 코드베이스가 단순해져 위협 모델·테스트 커버리지가 명확해진다.

### 부정적
- 명세상 빌링 항목 일부(예약·정액·크레딧·쿠폰)가 미구현으로 남는다. **이는 단점이 아닌 의도이며, 본 ADR 자체가 그 의도의 근거**임을 README와 scope.md에서 명시한다.
- 미래에 위 항목이 필요해질 경우, 본 ADR을 **Superseded** 처리한 신규 ADR이 필요하다.

### 중립적
- PG 어댑터 인터페이스를 잘 정의해 두면, 미래에 실제 PG 연동을 붙이는 것은 코드 변경 범위가 작다.

## Operational Implications — 운영 영향

- 빌링 관련 권한(가격 정책 수정·청구서 발행·미터링 read)을 **각각 분리된 역할**로 정의한다 → IAM ADR에서 반영.
- 청구서 PDF는 **생성 즉시 별도 보관소**(예: append-only 오브젝트 스토리지)에 저장하고, 본 데이터베이스에는 메타데이터·해시만 보관.
- 단가 변경은 항상 **변경 이력 테이블**에 함께 기록된다. 단가 테이블 직접 UPDATE는 금지되며, 변경은 항상 신규 레코드 + 유효기간 형식.

- 빌링 데이터의 보관 기간(법정 보관 의무 — 전자상거래법 5년 등) → 컴플라이언스 매핑 문서와 함께 확정.

## 본 결정의 회귀 트리거

다음 중 하나가 발생하면 본 ADR을 재검토한다.

- 실제 사업화 의사결정으로 PG 실연동이 필요해짐 → 신규 ADR로 크레딧 또는 직접 결제 흐름 도입 검토.
- 실 사용자·운영자 피드백에서 "다중 빌링 모델 부재"가 핵심 결격 사유로 반복 지적됨 → 신규 ADR로 구독형 정액 과금 또는 예약 모델 추가 검토.
