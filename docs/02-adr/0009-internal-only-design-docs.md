# ADR-0009: 일부 설계 산출물을 내부 전용으로 분리한다

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-05-28
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0001 (ADR 운영 체계)

## Context — 맥락

본 프로젝트는 공개 개발 리포로 운영된다 (라이선스는 Phase 1 진입 시점에 별도 ADR로 결정). 그러나 다음 두 설계 산출물은 외부 공개의 가치가 낮거나, 외부 공개를 위해서는 별도의 정제 작업이 필요하다.

1. **`CLAUDE.md` (프로젝트 헌법 — 에이전트 운영 지침)**
   - 매 세션 시작 시 자동 로드되는 작업 규칙·금지 사항·읽기 순서를 담는다.
   - 사용 도구(AI 에이전트)와 내부 운영 흐름을 직접적으로 노출한다.
   - 외부 기여자에게는 README → vision → scope → ROADMAP 순서가 자연스러우며, CLAUDE.md를 따로 노출할 가치가 낮다.

2. **`docs/00-overview/requirements.md` (요구사항 추적표)**
   - 원본 명세에서 파생된 FR/NFR 추적표. 명세 항목 ID·상태·Phase·사유·ADR을 한 표로 통합.
   - 원본 명세 자체가 외부에 공개되지 않은 상태이므로, 추적표의 항목명만 외부에 노출하면 맥락이 비어 오해를 부른다.
   - 외부 공개를 하려면 명세 항목 ID·맥락을 정제한 별도 요약본이 필요하다 (즉시 필요하지 않음).

또한 두 파일은 다른 외부 공개 문서에서 링크되면 깨진 링크가 된다. 예: 이전에 `README.md`의 "빠른 둘러보기" 표에 `requirements.md` 행이 있었음.

## Decision — 결정

위 두 파일을 **내부 전용**으로 유지하고 다음 정책을 적용한다.

1. **`.gitignore` 등록**: `CLAUDE.md`와 `docs/00-overview/requirements.md` 두 파일을 `.gitignore`에 명시적으로 추가한다. 파일명은 그대로 유지한다(별도 디렉터리·접미사 이동 없이).
2. **외부 공개 문서에서 링크·언급 금지**: 외부에 공개되는 추적 파일(README·vision·scope·ADR 등)에서 두 파일을 링크하거나 파일명을 노출하지 않는다. 단, **본 ADR과 `.gitignore`처럼 정책 자체를 정의·집행하기 위해 대상 파일명을 명시해야 하는 경우는 예외**다.
3. **결정 로그 등에서 파일명 노출 시 일반화**: `docs/90-meta/decision-log.md`처럼 외부에 공개되는 결정 로그에서 두 파일을 언급해야 할 때는 "내부 요구사항 추적표", "내부 에이전트 지침" 등 일반화된 표현을 사용한다.
4. **로컬 파일 자체는 그대로 유지**: 두 파일은 작성자·에이전트가 계속 활용하므로 로컬 작업 트리에서는 정상 보유한다.

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **모두 공개** (현 위치 그대로 리포에 포함) | 100% 투명성, 별도 관리 부담 없음 | 외부 공개 가치 낮음, 원본 명세 비공개 상태에서 추적표만 공개 시 맥락 결손, 에이전트 운영 흐름 노출 | 거부 |
| **별도 비공개 디렉터리(`docs/private/`)로 이동** | 비공개 영역의 위치가 시각적으로 명확 | 파일명·경로 변경 비용, CLAUDE.md는 루트에 있어야 자동 로드되는 도구 관습과 충돌, 두 파일의 자연스러운 위치를 흔듦 | 거부 |
| **별도 접미사(`*.private.md`) 채택** | 기존 `.gitignore` 패턴(`*.private.md`)을 그대로 활용 가능 | 파일명 자체가 변경되어 내부 참조·도구 관습이 깨짐, 외부에서는 어차피 안 보이므로 접미사가 가치 적음 | 거부 |
| **현 파일명 유지 + 명시적 `.gitignore` 등록** | 변경 비용 최소, 의도가 `.gitignore` 본문에 직접 명시되어 명확 | 두 파일을 외부 추적 문서에서 링크하지 않을 규율이 필요 (정책으로 강제) | **채택** |
| **공개용 요약본을 즉시 분리** | 외부 기여자에게도 요구사항 가시화 | 추적표 정제 작업량 큼, 현 시점에 외부 기여자 수요가 명확하지 않음 | 보류 (Phase 1 진입 시 필요해지면 별도 ADR로 결정) |

## Consequences — 결과

### 긍정적
- 외부 공개 영역에서 운영 메타데이터(에이전트 지침·원본 명세 파생물) 노이즈가 제거된다.
- 보안·운영 노출 면적이 좁아진다.
- 내부 추적성은 그대로 유지된다 (두 파일이 로컬에 그대로 존재).

### 부정적
- 외부 추적 문서에서 두 파일을 링크하지 않을 규율이 필요. 위반 시 외부에서는 깨진 링크가 된다.
- `decision-log.md` 등에서 작업을 기록할 때 파일명을 노출하지 않도록 표현을 일반화해야 한다 (이중 어휘 부담).
- 새 기여자가 합류해도 두 파일을 보지 못하므로, CLAUDE.md의 자동 로드 효과가 그 사람에게는 적용되지 않는다 (당분간 1인 운영 전제하 무영향).

### 중립적
- 향후 외부 공개용 요약본 분리가 필요해지면 본 ADR을 Superseded 처리하거나 보완 ADR을 추가한다.
- 두 파일 외에 새로운 내부 전용 산출물이 생기면 본 ADR의 정책을 그대로 확장 적용한다 (별도 ADR 불필요).

## Operational Implications — 운영 영향

- 외부 추적 문서를 작성·갱신할 때마다 두 파일에 대한 링크·파일명 노출 여부를 점검한다.
- 새 산출물이 "원본 명세에서 직접 파생" 또는 "에이전트 운영 지침"에 해당하면 본 ADR의 정책 대상으로 검토한다.
- `git status`에서 두 파일이 다시 staged 되는 일이 없는지 주기적으로 확인 (`.gitignore` 등록만으로 충분하지만 만약 `-f` 강제 추가 시 경로 유출 가능).

## Open Questions

- 외부 공개용 요약본을 언제 분리할 것인가? → 외부 기여자 수요·Phase 1 진입 시점에 재검토.
- 본 정책 대상이 늘어날 경우 별도 분류 디렉터리가 필요한지 → 대상이 3개 이상이 되면 재검토.
