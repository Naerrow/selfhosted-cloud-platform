## Context — 맥락

본 플랫폼은 Proxmox 자체 클러스터링을 사용하지 않으므로(ADR-0003) 테넌트 격리는 본 플랫폼이 단독으로 책임진다. 격리 실패는 본 프로젝트의 가치 명제 "운영·보안·감사 가능성의 깊이"를 정면 부정한다.

격리 책임은 네 층에 걸쳐 있다.

1. **컴퓨팅**: VM/LXC 인스턴스의 테넌트 귀속, 노드 공유 시 사이드채널 위험.
2. **네트워크**: 테넌트 간 L2/L3 격리. Proxmox는 SDN을 직접 강제하지 않는다.
3. **스토리지**: 볼륨·이미지·스냅샷·백업의 테넌트 귀속 + 쿼터.
4. **메타데이터·API**: 모든 컨트롤 플레인 쿼리에 테넌트 ID 필터, 우회 가능 경로 부재.

본 프로젝트는 ADR-0001의 "비자명한 결정" 기준 중 **보안에 영향을 주는 결정**에 해당하므로 ADR 필수. CSAP 매핑 문서·STRIDE 위협 모델과 직접 연결되며, Phase 2 (IAM & Multi-tenancy) 진입의 핵심 제약.

페르소나 측면: P-EU → P-TA, P-TA → P-CSP의 권한 상승(Elevation of Privilege) 시나리오를 항상 가정한다 (`personas.md` §7).

## Decision — 결정

[Phase 1 진입 전 확정.]

본 ADR이 다룰 결정 항목 (확정 시 본 섹션에 채워질 내용):

- **데이터 모델 수준 강제**: 모든 자원 테이블에 `tenant_id NOT NULL` + 모든 쿼리에 테넌트 필터를 ORM·미들웨어 어느 층에서 강제하는가?
- **네트워크 격리 메커니즘**: VLAN / VXLAN / EVPN / OVS 어느 조합인가? Phase 1에서 강제는 어디까지인가?
- **스토리지 격리**: 노드별 LVM/ZFS 풀에서 테넌트 식별을 메타데이터로만 할 것인가, 풀 자체를 분리할 것인가?
- **메타데이터 격리**: API 게이트웨이 + 인가 미들웨어 + 쿼리 빌더 3중 방어인가?
- **테스트 강제**: 테넌트 격리 회귀 테스트 (악의적 테넌트 시뮬레이션) 도입 형태.

## Open Questions

- Phase 1 (단일 테넌트 가정)에서 격리 강제 수준은 어디까지인가? "단일 테넌트만 가입 가능" + "쿼리는 이미 테넌트 필터" 정도인가?
- 메타데이터 격리 우회 발견 시 즉시 정지하는 fail-closed 정책이 코드 레벨에서 어떻게 강제되는가?
- 노드 자체가 침해될 때 테넌트 자원이 어디까지 노출되는가 (위협 모델 v0와 연결)?
- 결정의 회귀 트리거: Phase 5 도입 시 새 자원 종류(LB/DNS/Object Storage)가 본 격리 모델로 일관되게 확장 가능한가?
