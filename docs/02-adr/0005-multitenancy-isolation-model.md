# ADR-0005: 멀티테넌시 격리 모델

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-06-08
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0002 (Proxmox 채택), 0003 (자체 오케스트레이션), 0004 (Go + sqlc + PostgreSQL 단일)

## Context — 맥락

본 플랫폼은 Proxmox 자체 클러스터링을 사용하지 않으므로(ADR-0003) 테넌트 격리는 본 플랫폼이 단독으로 책임진다. 격리 실패는 본 프로젝트의 가치 명제 "운영·보안·감사 가능성의 깊이"를 정면 부정한다.

격리 책임은 네 층에 걸쳐 있다.

1. **컴퓨팅**: VM/LXC 인스턴스의 테넌트 귀속, 노드 공유 시 사이드채널 위험.
2. **네트워크**: 테넌트 간 L2/L3 격리. Proxmox는 SDN을 직접 강제하지 않는다.
3. **스토리지**: 볼륨·이미지·스냅샷·백업의 테넌트 귀속 + 쿼터.
4. **메타데이터·API**: 모든 컨트롤 플레인 쿼리에 테넌트 ID 필터, 우회 가능 경로 부재.

본 프로젝트는 ADR-0001의 "비자명한 결정" 기준 중 **보안에 영향을 주는 결정**에 해당하므로 ADR 필수. CSAP 매핑 문서·STRIDE 위협 모델과 직접 연결되며, Phase 2 (IAM & Multi-tenancy) 진입의 핵심 제약.

페르소나 측면: P-EU → P-TA, P-TA → P-CSP의 권한 상승(Elevation of Privilege) 시나리오를 항상 가정한다 (`personas.md` §7).

## Decision — 결정

4층 격리를 다음과 같이 확정한다. ADR-0004(PostgreSQL 단일 + sqlc)를 전제로 한다.

1. **메타데이터 격리 (DB 계층) — 이중 방어**:
   - 모든 자원 테이블에 `tenant_id NOT NULL`.
   - **PostgreSQL Row-Level Security(RLS)** 정책으로 DB 엔진 레벨에서 테넌트 필터 강제. 세션 변수(`SET app.current_tenant`)에 인가된 테넌트 ID를 주입하고, RLS 정책이 모든 행 접근을 그 값으로 제한한다.
   - **추가로** 앱 계층의 sqlc 쿼리도 명시적으로 `WHERE tenant_id = $n`을 둔다. RLS(DB 강제)와 앱 쿼리 필터(코드 리뷰 가시성)의 **이중 방어** — 어느 한 층의 실수가 즉시 전체 노출로 가지 않게 한다.
2. **네트워크 격리**: Phase 1은 **단일 테넌트 가정**이므로 네트워크 강제는 최소(기본 분리)로 둔다. 테넌트별 오버레이(VXLAN/OVN)는 **Phase 5 네트워킹 ADR**에서 본격 설계한다(ADR-0003 Operational Implications와 정합).
3. **스토리지 격리**: 노드별 LVM/ZFS 풀을 **공유**하되, 볼륨·이미지·스냅샷을 **`tenant_id` 메타데이터로 귀속** + 테넌트별 **쿼터**를 강제한다. 네이밍 규칙으로 테넌트 식별을 명시화한다. 풀 자체 분리는 혼합 HW·미니 PC 환경의 용량 효율을 해치므로 채택하지 않는다.
4. **fail-closed 정책**: 인가된 테넌트 컨텍스트(세션 변수)가 없으면 **쿼리가 0행을 반환하거나 거부**되도록 RLS 기본 정책을 deny로 둔다. 테넌트 컨텍스트 주입은 API 게이트웨이의 인가 미들웨어가 책임지며, 주입 실패 시 요청을 거부한다(미들웨어 → RLS 2단).
5. **테스트 강제**: Phase 1은 기본 테넌트 필터 단위 테스트. **악의적 테넌트 시뮬레이션(타 테넌트 자원 접근 시도) 회귀 테스트는 Phase 2(멀티테넌시) 진입 시 CI 게이트로 강제**한다.

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **단일 DB + tenant_id + RLS + 앱 필터 (이중 방어)** | 운영 단순(1인 환경·ADR-0004 단일 PostgreSQL과 정합), DB 강제 + 코드 가시성 이중 방어, sqlc 명시 SQL과 정합 | RLS 정책 누락 시 위험 → 앱 필터로 보완 | **채택** |
| 단일 DB + tenant_id + 앱 필터만 (RLS 없음) | 구현 가장 단순 | 방어 1중 — 앱 버그 하나가 전체 노출(R-010 치명적) | 거부 — 단일 방어는 깊이 가치와 충돌 |
| 테넌트별 스키마 분리 | 격리 강함, 백업 단순 | 테넌트 증가 시 마이그레이션 N배, 1인 운영 부담 폭증 | 거부 — 운영 부담이 1인 범위 초과 |
| 테넌트별 DB 인스턴스 | 격리 최강 | 인스턴스 폭증, 미니 PC 환경 부적합 | 거부 — 운영·자원 부담 과함 |
| VLAN per tenant (수동) | 명시적·검증 쉬움 | VLAN ID 4094 한계·동적 할당 어려움, Phase 1 과잉 | 거부 (네트워크는 Phase 5로 이연) |
| VXLAN/OVN per tenant | 확장성·자동화 | 컨트롤 플레인 복잡도 큼, Phase 1 시연 범위 초과 | **Phase 5로 이연** (거부 아님) |

## Consequences — 결과

### 긍정적
- **이중 방어**로 한 층(앱 또는 DB)의 실수가 즉시 전체 테넌트 노출로 가지 않는다 — R-010(치명적) 완화의 핵심.
- 단일 PostgreSQL + RLS는 ADR-0004와 정합해 1인 운영 부담이 최소.
- sqlc의 명시적 `WHERE tenant_id`는 코드 리뷰에서 필터 누락을 눈으로 잡을 수 있다(위협 모델 TH-06 통제).
- 스토리지 공유 풀 + 쿼터는 혼합 HW·미니 PC의 용량 효율을 지킨다.

### 부정적
- RLS 세션 변수 주입을 모든 요청 경로에서 빠짐없이 해야 한다 — 누락 시 fail-closed(0행)로 안전하나, 정상 기능이 막히는 버그로 나타날 수 있다. 통합 테스트로 검증.
- 네트워크 강한 격리를 Phase 5로 미루므로, Phase 1~4의 다중 테넌트 네트워크 격리는 **약한 상태**임을 명시해야 한다(위협 모델·CSAP 매핑에 한계로 기록).
- Phase 2의 악의적 테넌트 회귀 테스트 전까지는 격리 강제가 코드 리뷰·단위 테스트 수준에 의존.

### 중립적
- 본 모델은 위협 모델 **TB-2(TH-06~08)** 의 통제를 구체화한다. 위협 모델 v0의 "TB-2를 4층으로 분해" 갱신 트리거가 본 ADR로 충족된다.

## Open Questions

> 핵심 결정(4층 + 이중 방어 + fail-closed + 테스트 시점)은 위에서 확정됐다. 남은 항목은 후속 Phase 세부다.

- RLS 세션 변수 주입을 미들웨어에서 강제하는 구체 패턴(커넥션 풀 재사용 시 변수 누수 방지) — Phase 1 구현 착수 시 확정.
- 노드 자체가 침해될 때 테넌트 자원 노출 범위(위협 모델 TH-11) — 네트워크 격리(Phase 5)·컴퓨팅 사이드채널과 함께 Phase 5에서 심화.
- 결정의 회귀 트리거: Phase 5 도입 시 새 자원 종류(LB/DNS/Object Storage)가 본 격리 모델(tenant_id + RLS + 쿼터)로 일관되게 확장 가능한가?
