# ADR-0001: 아키텍처 의사결정을 ADR로 기록한다

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-05-25
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 없음 (최초)

## Context — 맥락

본 프로젝트는 **공개 개발 리포**로 운영되며 (라이선스는 Phase 1 진입 시점에 별도 ADR로 결정), 설계 의사결정의 근거와 흐름은 코드만큼 중요한 산출물이다. 또한 1인 운영 + 장기 프로젝트 + 외부 보조 도구(AI 등) 사용이라는 조건에서 **컨텍스트가 무너지지 않게** 결정 사항을 한 곳에 응집해 둘 필요가 있다.

코드 주석·이슈·PR 설명만으로는 의사결정의 "이유"가 시간이 지나며 산일된다. 결정의 흔적이 어디에 어떻게 남는지 일관된 규칙이 없으면, 미래의 자신도 결정 사유를 재구성할 수 없게 된다.

## Decision — 결정

본 프로젝트의 **모든 비자명한 설계 의사결정**은 ADR(Architecture Decision Record) 형식으로 `docs/02-adr/` 디렉터리에 기록한다.

### 형식

각 ADR은 다음 섹션을 갖는다.

1. **제목 헤더**: `ADR-NNNN: 한 줄 결정 요약` (현재형 진술)
2. **메타블록**: 상태, 결정일, 결정자, 관련 ADR
3. **Context**: 결정이 필요해진 배경
4. **Decision**: 우리가 무엇을 결정했는가 (현재형, 단정형)
5. **Alternatives Considered**: 비교한 대안과 각각의 평가
6. **Consequences**: 이 결정의 긍정적·부정적·중립적 결과
7. **Open Questions** (선택): 미해결 항목과 향후 트리거

### 상태 라이프사이클

- **Proposed**: 작성되었으나 아직 채택되지 않음
- **Accepted**: 채택되어 효력 있음
- **Deprecated**: 더 이상 권장하지 않으나 명시적 대체는 없음
- **Superseded by ADR-NNNN**: 다른 ADR로 명시적 대체됨

상태 전이는 ADR 본문에 항상 명시한다. **Accepted된 ADR의 변경은 ADR-0010의 4단계 분류 체계를 따른다**:

- **분류 1·2** (오타·서식·깨진 링크 / 1차 출처 사실 정정 중 결정 영향 없음): 본문 직접 정정 가능.
- **분류 3** (1차 출처 사실 정정 중 결정 근거·결론에 영향 있음): 신규 ADR을 작성하고 기존 ADR의 상태를 Superseded로 갱신.
- **분류 4** (운영 정책의 신설·변경): 신규 ADR로 다룬다.

판단 기준과 절차는 [ADR-0010](./0010-research-doc-correction-exception.md)을 따른다.

### 번호 부여

- 4자리 패딩 (0001, 0002, …)
- 번호는 단조 증가하며 재사용하지 않는다
- 주제별로 미리 번호를 할당해도 무방하다. 인덱스의 상태가 권위를 갖는다.

### 명명 규칙

파일명: `NNNN-kebab-case-summary.md` (영어, 명령형 동사 권장)
예) `0002-use-proxmox-as-hypervisor.md`

### "비자명한 결정"의 기준

다음 중 **하나라도** 해당하면 ADR을 작성한다.

- 되돌리기 비용이 큰 결정 (언어·프레임워크·데이터 모델·인증 방식 등)
- 비교 가능한 대안이 2개 이상 존재했던 결정
- 보안·법규·운영에 영향을 주는 결정
- "이거 왜 이렇게 했지?"를 한 달 뒤의 자신이 물을 가능성이 있는 결정
- 명세상 기능을 **제외**하기로 한 결정 (이게 특히 중요)

## Alternatives Considered — 대안 비교

| 대안 | 평가 |
|---|---|
| 결정 사항을 코드 주석/PR 설명에만 남긴다 | 분산되어 추적 불가, 시간 경과 후 재구성 불가능 |
| 위키(Confluence 등) 별도 도구 사용 | 코드와 분리되어 동기화 부담, 셀프호스팅 오픈소스 프로젝트에는 부적합 |
| ADR (Michael Nygard 포맷) | 코드와 한 리포에 공존, 깃 이력으로 결정의 흐름 추적 가능, 외부 기여자·운영자 접근성 우수 → **채택** |
| MADR (Markdown Architectural Decision Records) | Nygard 포맷의 확장. 더 구조화되어 있으나 1인 프로젝트엔 과한 측면. 필요 시 향후 마이그레이션 가능 |

## Consequences — 결과

### 긍정적
- 모든 결정의 이유가 한 곳에 남는다.
- 외부 기여자나 운영자가 ADR 인덱스 하나만 보고 프로젝트의 사고 흐름을 파악할 수 있다.
- 미래의 자신이 컨텍스트를 빠르게 복원할 수 있다.
- 결정의 번복 이력(Superseded)이 학습 자산이 된다.

### 부정적
- 문서 작성 오버헤드가 발생한다.
- ADR 인플레이션 (사소한 결정까지 ADR화)의 위험이 있다 → "비자명한 결정 기준"으로 완화.

### 중립적
- ADR은 코드 변경과 함께 동일 PR에서 갱신해야 한다 (별도 워크플로우 필요).
- **첫 공개 커밋 이후 Phase 0 (설계만 진행, 코드 0줄) 변경에 대한 예외**: 코드·PR 흐름이 아직 존재하지 않으므로, ADR 갱신은 `docs/90-meta/decision-log.md` 항목과 동일 단위로 묶어 추적한다. 첫 공개 커밋 이전의 작업은 첫 커밋의 원본으로 통합되며 별도 추적하지 않는다 (`decision-log.md` 기록 정책 참조). Phase 1 진입 시점부터 본 절의 PR 동기화 원칙이 효력 발생.

## Open Questions

- ADR 자동화 도구(adr-tools 등) 도입 여부 → Phase 1 진입 시점에 재검토.
- ADR 변경 시 PR 템플릿에 ADR 갱신 체크박스 추가 → CI/CD 뼈대 작성 시 함께 적용.
