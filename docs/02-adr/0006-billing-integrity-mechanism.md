# ADR-0006: 빌링 데이터 무결성 보장 방식

- **상태(Status)**: Accepted
- **결정일(Date)**: 2026-06-17
- **결정자(Decider)**: 프로젝트 메인테이너
- **관련 ADR**: 0008 (Phase 3 빌링 스코프), 0004 (Go + sqlc + PostgreSQL 단일), 0005 (멀티테넌시 격리)

## Context — 맥락

ADR-0008은 빌링을 "결제 기능이 아닌 감사 대상 시스템"으로 다룬다고 명시했고, 그중 무결성 보장 메커니즘의 구체화를 본 ADR에 위임했다. 또한 내부 에이전트 지침의 빌링 코드 작성 규칙과 R-006(리스크 레지스터)은 빌링 무결성 결함을 **치명적** 영향으로 분류한다.

본 ADR은 다음을 결정한다.

- **append-only 미터링의 물리적 강제 방식**: 단순 DB 트리거 / WORM 스토리지 / 이벤트 로그 분리 중 어느 것인가?
- **무결성 증명 자료 구조**: 해시 체인(블록체인 유사) vs 머클 트리 vs 둘 다.
- **검증 가능성**: 외부 감사자가 청구서 단일 발행본만 보고 해당 기간 미터링·요율·합계의 무결성을 재계산 가능한가?
- **Idempotency 정의**: 같은 (테넌트, 기간) 입력에 대해 같은 청구서 해시가 재생산되는 함수 정의.
- **빌링 감사 로그 분리 보관**: 일반 운영 로그와 다른 스토리지·접근 제어.

본 ADR은 Phase 3 빌링 엔진 구현이 시작되기 전에 확정되어야 한다. Phase 1·2 단계에서는 본 ADR의 결정이 데이터 모델 설계 (Phase 1 기본 미터링)에 미리 영향을 준다.

## Decision — 결정

빌링 무결성 메커니즘을 다음과 같이 확정한다. ADR-0004(단일 PostgreSQL + sqlc)·ADR-0008(빌링 깊이 요구)을 전제로 한다.

1. **무결성 증명 자료구조 — 해시 체인 + 기간 머클 루트 (조합)**:
   - **개별 미터링 레코드**는 **해시 체인**으로 묶는다. 레코드 `r_n`은 `hash_n = H(canonical(r_n.payload) || hash_{n-1})`을 저장한다(테넌트별 체인, `H` = SHA-256). 한 레코드의 변조·삭제·순서 변경이 이후 모든 해시를 깨뜨리므로 **append-only 성질과 순서 보증**을 동시에 강제한다.
   - **청구서(기간 단위 발행본)**는 해당 (테넌트, 기간)에 속한 미터링 레코드 해시들로 **머클 트리**를 구성하고, **머클 루트**를 청구서에 임베드한다. 외부 감사자는 청구서에 첨부된 루트 + 머클 증명(proof)으로 **일부 레코드만 받아도** 해당 레코드가 그 기간 집계에 포함됐음을 재계산·검증할 수 있다.
   - 즉 **해시 체인은 "원천 데이터의 무결성·순서"**, **머클 루트는 "기간 청구서의 부분 검증 가능성"** 을 담당한다. 두 구조는 같은 레코드 해시(`hash_n`)를 입력으로 공유한다.
2. **append-only의 물리적 강제 (ADR-0004 단일 PostgreSQL 전제)**:
   - 미터링 원천 테이블은 **INSERT만 허용**한다. 애플리케이션 DB 역할에 해당 테이블의 `UPDATE`/`DELETE` 권한을 부여하지 않고(GRANT 수준 강제), 추가로 `BEFORE UPDATE OR DELETE` 트리거가 예외를 던지는 **이중 차단**을 둔다(ADR-0005의 이중 방어 원칙과 동형).
   - 외부 WORM 스토리지·전용 이벤트 로그 제품은 도입하지 않는다 — 1인 운영 부담과 ADR-0004의 단일 PostgreSQL 결정에 어긋난다. WORM 성질은 위 권한+트리거+해시 체인의 조합으로 논리적으로 달성한다.
   - **정정은 보정 레코드(correction record)로만** 한다. 보정 레코드는 원 레코드를 가리키는 `corrects_id`를 갖고, 그 자신도 해시 체인에 새 레코드로 append된다. 원 레코드는 절대 변경되지 않는다(ADR-0008 깊이 요구 1).
3. **Idempotency 정의**:
   - 청구서 생성은 순수 함수 `invoice = f(tenant_id, period, 해당 기간의 미터링 레코드 집합, 적용 단가 버전)` 으로 정의한다. 같은 입력은 **같은 청구서 해시(머클 루트 포함)** 를 재생산한다.
   - 청구서 재발행은 **새 발행본을 만들지 않고** 동일 해시를 재확인한다(ADR-0008 깊이 요구 4). 입력 집합이 달라졌다면(예: 보정 레코드 추가) 그것은 재발행이 아니라 **신규 정정 청구서**이며, 원 청구서를 가리키는 링크를 둔다.
   - 단가는 직접 UPDATE하지 않고 **신규 레코드 + 유효기간** 방식이므로(ADR-0008 운영 영향), `f`의 "적용 단가 버전"은 기간 시점으로 결정론적으로 고정된다.
4. **빌링 감사 로그 분리 보관**:
   - 빌링 감사 로그(단가 변경·수동 조정·청구서 발행/재확인)는 **운영 로그와 다른 PostgreSQL 스키마 + 별도 권한**에 저장한다(ADR-0004의 감사 별도 스키마 결정과 동일 구조). 일반 운영 역할은 이 스키마에 write할 수 없다.
   - 감사 로그 자체도 append-only(권한+트리거)이며, 그 위에 해시 체인을 적용해 변조를 탐지한다.
   - 청구서 PDF 발행본은 본 DB에 **메타데이터·해시(머클 루트)만** 두고, 발행본 자체는 별도 보관소(append-only 오브젝트 스토리지)에 저장한다(ADR-0008 운영 영향). 보관소 구체 선택은 Phase 3 구현 시점에 결정한다.

## Alternatives Considered — 대안 비교

| 대안 | 강점 | 약점 | 결정 |
|---|---|---|---|
| **해시 체인(개별 레코드) + 머클 루트(기간 청구서)** | 원천 데이터 순서 보증 + 청구서 부분 검증 가능성을 둘 다 확보. "청구서 단일 발행본으로 외부 감사 재계산"(본 ADR Context) 요구 충족 | 운영 복잡도 ↑(두 구조 동시 유지) | **채택** |
| 해시 체인 단독 | 구현·검증 선형, 1인 운영 부담 최소 | 부분 검증 시 체인 전체 필요 → 외부 감사자가 청구서 한 장만으로 부분 재계산 불가 | 거부 — 외부 감사 부분 검증 요구 미충족 |
| 머클 트리 단독 (기간 루트) | 부분 검증·증명 효율 | 개별 레코드의 append-순서·중간 삽입 탐지가 약함 | 거부 — 원천 데이터 순서 무결성 보증 부족 |
| 외부 timestamping (RFC 3161 / OpenTimestamps) | 외부 시점 보증(제3자 신뢰) | 외부 서비스 의존 → 셀프호스팅·데이터 주권 가치(vision §2)와 충돌 | 거부 (회귀 트리거로 유지 — Open Questions) |
| 단순 DB append + 정기 백업 해시 | 가장 단순 | 위변조 탐지가 약하고 실시간 무결성 보장 없음 | 거부 (R-006 치명적) |

## Consequences — 결과

### 긍정적
- **외부 감사자가 청구서 단일 발행본(+머클 증명)만으로** 그 기간 미터링·집계의 무결성을 재계산할 수 있다 — 빌링을 "감사 대상 시스템"으로 다루는 vision §3.2·ADR-0008의 핵심 요구를 충족한다.
- 해시 체인이 미터링 원천의 **변조·삭제·순서 변경을 모두 탐지**하므로 R-006(치명적)의 핵심 완화책이 된다.
- 권한(GRANT)+트리거+해시 체인의 **다층 강제**로, 한 층(예: 트리거 누락)의 실수가 즉시 무결성 붕괴로 가지 않는다(ADR-0005 이중 방어와 동형).
- 외부 WORM·timestamping 서비스 없이 **단일 PostgreSQL 안에서** 달성하므로 ADR-0004·1인 운영 부담과 정합한다.

### 부정적
- 해시 체인과 머클 트리 **두 구조를 동시에 유지**해야 한다 → 미터링 적재·청구서 생성 코드가 복잡해지고, 두 구조의 입력(레코드 해시)이 일관되게 공유되도록 테스트로 강제해야 한다.
- 해시 체인은 테넌트별 **순차 append**를 전제하므로, 고동시성 적재 시 체인 말단 경합이 발생할 수 있다 → 테넌트별 체인 분리·배치 적재로 1차 대응(Phase 3 미터링 파이프라인 설계 과제).
- 머클 증명을 청구서 PDF에 어떻게 임베드할지(부록/별도 검증 파일)는 구현 세부로 남는다(Open Questions).

### 중립적
- 본 결정은 위협 모델 **TB-6(TH-16·20)** 의 통제를 구체화한다 — TH-16(미터링 변조)은 해시 체인, TH-20(청구서 비재현)은 idempotency 정의로 확정된다. 위협 모델 v0의 "ADR-0006 Accepted 시 무결성 자료구조·검증 절차로 구체화" 트리거가 본 ADR로 충족된다.
- 빌링 감사 로그 보관 기간은 한국 법정 보관 의무(전자상거래법 등)와 맞물리며, 그 구체 연한은 **CSAP·개인정보보호법 매핑 문서에서 1차 출처 확인 후 확정**한다(본 ADR은 자료구조·강제 방식까지를 다룬다).

## Open Questions

> 핵심 결정(자료구조 = 해시 체인 + 머클 루트, append-only 강제 = 권한+트리거, idempotency 정의, 감사 로그 분리)은 위에서 확정됐다. 남은 항목은 Phase 3 구현 세부와 후속 트리거다.

- 청구서 PDF에 머클 증명을 어떻게 임베드하는가(별도 부록 PDF / 검증용 JSON 동봉 / 청구서 본문 해시 표기) — Phase 3 청구서 생성기 설계 시 확정.
- 보정 레코드는 본 ADR에서 `corrects_id` + 해시 체인 append로 확정됐다. 원 레코드 → 보정 레코드 **역방향 조회**를 인덱스로 둘지는 Phase 3 데이터 모델 세부.
- 빌링 감사 로그·청구서의 **법정 보관 기간**(전자상거래법 등) — 본 ADR은 자료구조까지를 다루고, 구체 연한은 **CSAP·개인정보보호법 매핑 문서에서 1차 출처(law.go.kr) 재확인 후 확정**한다(작성일 2026-05-25 사전 조사 자료는 진행 시점 재확인 대상).
- 해시 체인 `H`로 SHA-256을 택했으나, 양자내성 등 장기 보관 요구 시 해시 알고리즘 마이그레이션 경로(이중 해시 기간) — Phase 3 중반 재검토.
- **회귀 트리거 (외부 timestamping)**: 외부 감사자가 "셀프 보관 해시는 운영자가 사후 위조 가능"을 문제삼으면, 거부했던 RFC 3161 / OpenTimestamps를 **머클 루트에만** 선택적으로 적용하는 신규 ADR을 검토한다(데이터 주권 가치와의 trade-off 재평가).
- **회귀 트리거 (Cloud Bursting)**: Phase 7 도입 시 AWS Cost Report 데이터의 무결성을 본 메커니즘에 어떻게 합치는가(외부 데이터의 해시 체인 편입 vs 별도 증명) — Phase 7 진입 시 재검토.
