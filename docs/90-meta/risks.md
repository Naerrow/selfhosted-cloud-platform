# Risk Register

> 본 문서는 프로젝트 진행 중 발생할 수 있는 리스크를 사전 식별·추적한다.
> 매 Phase 종료 시 재검토하며, 새로 발견한 리스크는 즉시 추가한다.

## 평가 척도

- **가능성(Likelihood)**: 낮음 / 중간 / 높음
- **영향(Impact)**: 낮음 / 중간 / 높음 / 치명적
- **우선순위(Priority)**: 가능성 × 영향의 정성 평가

## 리스크 목록

### R-001. 스코프 크리프 (Scope Creep)
- **가능성**: 높음
- **영향**: 높음 (핵심 가치 명제 희석, 어느 영역도 깊게 다루지 못할 위험)
- **상황**: 명세 항목이 많고, 개별 항목이 모두 매력적으로 보임.
- **완화**: `scope.md`와 `vision.md`의 회귀 검증 질문을 모든 신규 기능 결정에 강제 적용. 새 기능 추가는 반드시 ADR 작성을 거침.
- **모니터링**: 매 Phase 종료 시 IN/OUT-OF-SCOPE 변경 이력 점검.

### R-002. 컨텍스트 무너짐
- **가능성**: 중간
- **영향**: 높음
- **상황**: 긴 프로젝트 + AI 도구 사용 + 1인 운영에서 결정 사유가 휘발될 가능성.
- **완화**: 모든 결정은 ADR 번호로 인용. **첫 공개 커밋 이후** `decision-log.md`에 매주 1건 이상 기록 (decision-log 기록 정책 본문 참조). 새 세션 시작 시 비전·스코프·ADR 인덱스·decision-log 최근 2주 + ROADMAP 요약을 함께 컨텍스트로 제공 (내부 에이전트 지침의 자동 로드/트리거 로드 분리 정책). 상세 ROADMAP·요구사항 추적표는 작업 성격에 따라 트리거 로드.
- **모니터링**: **첫 공개 커밋 이후** decision-log 공백이 2주 이상이면 경고.

### R-003. 할루시네이션·사실 오류
- **가능성**: 중간
- **영향**: 중간~높음 (프로젝트 신뢰도 훼손)
- **상황**: Proxmox API, AWS API, 한국 법규 등의 구체 사실에서 발생 가능.
- **완화**:
  - 1차 출처 인용 의무화 (공식 문서 URL + 확인 일자).
  - 사실 주장에 출처 없는 것은 `[REVIEW]` 태그를 본문에 남김.
  - 법규 항목은 작성일 명시 + 사업화 시 별도 검토 안내.
- **모니터링**: 매 Phase 종료 시 `[REVIEW]` 잔여 항목 점검.

### R-004. Proxmox API 호환성 변화
- **가능성**: 낮음
- **영향**: 중간
- **상황**: 본 플랫폼이 Proxmox REST API에 의존. API 변경 시 영향 가능.
- **완화**: Proxmox 어댑터 계층을 두어 변경 영향을 격리. 통합 테스트로 회귀 감지.
- **모니터링**: Proxmox 릴리스 노트 정기 확인.

### R-005. 한국 규제 변화
- **가능성**: 중간
- **영향**: 중간 (본 프로젝트 자체에는 영향 적음, 실제 사업화 시 큼)
- **상황**: CSAP 등급제, 개인정보보호법 국외이전 경로 등이 자주 갱신됨.
- **완화**: 컴플라이언스 매핑 문서에 작성일·근거 법령 명시. 사업화 의사결정 시 변호사 검토 안내를 README와 vision.md에 명시.
- **모니터링**: Phase 종료 시 매핑 문서 재확인.

### R-006. 빌링 무결성 결함
- **가능성**: 중간
- **영향**: 치명적 (감사 실패, 신뢰도 즉시 상실)
- **상황**: 무결성 검증 메커니즘 설계·구현 오류 가능.
- **완화**: ADR-0006(Accepted, 2026-06-17)에서 메커니즘 확정 — 해시 체인(개별 미터링 레코드) + 기간 머클 루트(청구서), append-only는 DB 권한 미부여+트리거로 물리 강제, 빌링 감사 로그 별도 스키마+권한 분리. 적대적 보안 리뷰(공격자 시나리오 시뮬레이션)로 사전 검증. idempotency·append-only를 테스트로 강제.
- **모니터링**: 매 Phase에서 빌링 관련 변경 시 적대적 보안 리뷰 의무화.

### R-007. 1인 번아웃·지속성
- **가능성**: 중간
- **영향**: 높음
- **상황**: 장기 프로젝트의 가장 현실적 위험.
- **완화**:
  - Phase 단위로 명확한 Exit 기준 + 데모 GIF → 매 단계 종료감각 확보.
  - 회고 MD 작성으로 진전 가시화.
  - 컷한 항목을 죄책감 없이 ADR로 명문화.
- **모니터링**: 2주 이상 진전 없으면 스코프 재검토 트리거.

### R-008. 라이선스·법적 이슈 (코드 공개)
- **가능성**: 낮음
- **영향**: 중간
- **상황**: AGPLv3 Proxmox 사용, 외부 OSS 의존성 라이선스 호환.
- **완화**: 본 프로젝트의 라이선스를 ADR로 별도 결정. CI에서 SBOM + 라이선스 스캔.
- **모니터링**: ADR-0011로 Apache-2.0 채택 결정 완료. 외부 의존성 추가 시 SBOM에 라이선스 호환 검증 의무(Phase 1 CI 게이트).

### R-009. 자체 오케스트레이션 책임 폭증 (Proxmox 클러스터 미사용 결과)
- **가능성**: 중간 (개발이 진행되면 자연 발생)
- **영향**: 높음 (장애 복구·상태 일관성·라이브 마이그레이션 모두 본 플랫폼이 직접 구현해야 함)
- **상황**: ADR-0003에서 Proxmox 자체 클러스터링(Corosync + pmxcfs + HA Manager)을 사용하지 않고 자체 오케스트레이션 레이어를 만들기로 결정. 이 결과로 HA 자동 페일오버, pmxcfs 설정 일관성, 라이브 마이그레이션을 본 플랫폼이 자체 구현해야 한다. 1인 운영 가능 범위와 직접 긴장 관계.
- **완화**:
  - Phase 4 이전까지는 자동 페일오버 대신 **알림만** 제공 (ADR-0003 결정).
  - 자체 메타데이터 DB를 단일 진실 공급원으로 둠 (ADR-0003 Consequences).
  - 라이브 마이그레이션은 Phase 4에서 별도 ADR로 다루며, 그 전까지 스코프에서 보류 (ADR-0003 Open Questions).
- **모니터링**: Phase 4 진입 시점에 본 리스크의 실현 정도(라이브 마이그레이션 요구·페일오버 자동화 요구의 강도)를 재평가.

### R-010. 멀티테넌시 격리 실패
- **가능성**: 중간
- **영향**: 치명적 (본 프로젝트 가치 명제 "운영·보안·감사 가능성의 깊이"를 정면 부정)
- **상황**: ADR-0003(Accepted)에서 Proxmox 자체 클러스터링을 사용하지 않기로 결정하면서 테넌트 격리 책임이 본 플랫폼에 단독으로 귀속되었다. 격리는 컴퓨팅·네트워크·스토리지·메타데이터 4층에 걸쳐 있고, 한 층의 우회 가능 경로가 전체 격리를 무력화한다. 구체 격리 모델은 ADR-0005(Accepted, 2026-06-08)에서 확정됐다 — 메타데이터=tenant_id+RLS+앱필터 이중 방어+fail-closed. 단 네트워크·컴퓨팅 강한 격리는 Phase 5로 이연돼 그 전까지 다중 테넌트 네트워크 격리는 **약한 상태**이며, Phase 1 단일 테넌트 가정과 Phase 2 강한 격리 강제 사이 retrofit 위험도 존재한다.
- **완화**:
  - ADR-0005에서 4층 격리 모델 확정(Accepted): 메타데이터 이중 방어(RLS+앱필터), 스토리지 메타귀속+쿼터, 네트워크는 Phase 5 이연.
  - 모든 자원 조회 쿼리에 테넌트 ID 필터 강제 + DB RLS 이중 방어는 ADR-0005에 반영.
  - 우회 가능 경로 발견 시 ADR-0005와 위협 모델 갱신 절차에 반영.
  - Phase 2 진입 시 테넌트 격리 회귀 테스트(악의적 테넌트 시뮬레이션)를 CI 게이트로 강제 — ADR-0005에서 확정.
- **모니터링**: 위협 모델 갱신 시 P-EU → P-TA → P-CSP → P-OP 권한 상승 시나리오 (personas.md §7) 점검. Phase 2 종료 시점에 회귀 테스트 통과 여부.

### R-011. 시크릿 유출·관리
- **가능성**: 중간
- **영향**: 치명적 (시크릿 유출 시 전 영역 횡적 이동 — 위협 모델 자산표상 시크릿은 최고 민감도)
- **상황**: ADR-0007(Accepted, 2026-06-23)에서 시크릿 저장을 **SOPS + age**로 확정했다. 새 외부 도구(SOPS·age) 도입 자체가 회귀 트리거다. 시크릿 종류(Proxmox Token·DB·JWT 키·OIDC·PG 자격증명)가 어댑터·API·빌링 3개 소비자에 걸쳐 있어, 한 소비자의 redact 누락(TH-21·22)이나 키 탈취(TH-23)가 전 영역 노출로 번질 수 있다. SOPS+age는 **시크릿 접근 단위 감사가 약하고**(파일 복호화 단위) **회전이 수동**이라는 한계를 ADR-0007이 명시했다. age 마스터 키 분실 시 전 시크릿 복호화 불가 위험도 운영자 단일 책임으로 남는다.
- **완화**:
  - ADR-0007에서 확정: 기동 시 복호화→**프로세스 메모리에만** 적재(평문 디스크·env 흩뿌림 금지), 시크릿 타입 redact 래퍼 + 구조화 로깅 allowlist(TH-21), 소비자별 provider 인터페이스로 접근 표면 제한(TH-22 — 단일 프로세스라 강한 격리 아닌 코드 규율, 강한 격리는 Phase 4+ 모듈 분리 재검토).
  - Proxmox Token 만료일+token-only 폐기, JWT 키 회전(Phase 1 폐기 절차→Phase 2 JWK kid 회전, TH-23).
  - 시크릿 유출 폐기·재발급 런북 Phase 1 최소 1건(`docs/04-operations/runbooks/`), age 마스터 키 백업 절차 포함.
  - 평문 시크릿은 git·로그·문서에 절대 금지(SOPS 설정·암호화 파일은 ADR-0009 내부 전용).
- **모니터링**: 강한 접근 감사·자동 회전 요구가 실질화되면 Vault류 도입을 신규 ADR로 재평가(ADR-0007 회귀 트리거). Phase 5 시크릿 종류 폭증 시 SOPS 파일 관리 부담 재평가.

## 회귀 트리거

다음 중 하나가 발생하면 본 문서를 즉시 갱신한다.
- 새 외부 의존성 도입
- 새 페르소나 등장
- 명세상 새 기능 검토
- Phase 종료 시점 (정기 갱신)
