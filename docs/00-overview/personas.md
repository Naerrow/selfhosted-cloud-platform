# Personas

> 본 문서는 시스템을 사용하는 주체를 페르소나로 정의한다.
> 모든 사용자 스토리, E2E 시나리오, IAM 권한 설계는 본 문서의 페르소나를 참조한다.

## 1. 페르소나 개요

본 플랫폼에는 4개 페르소나가 존재한다.

| 코드 | 명칭 | 역할 한 줄 요약 |
|---|---|---|
| **P-OP** | 플랫폼 운영자 (Platform Operator) | 본 플랫폼 자체를 설치·운영·업데이트·모니터링하는 시스템 관리자 |
| **P-CSP** | CSP 사업자 (Cloud Service Provider) | 본 플랫폼 위에서 클라우드 서비스를 자기 고객에게 판매·과금하는 사업 주체 |
| **P-TA** | 테넌트 관리자 (Tenant Admin) | P-CSP의 고객으로, 한 조직(테넌트) 안의 자원·사용자·예산을 책임지는 사람 |
| **P-EU** | 엔드 유저 (End User) | 테넌트 안에서 VM/스토리지 등을 실제로 생성·사용하는 개발자/엔지니어 |

> 참고: P-OP와 P-CSP는 소규모 환경(예: 1인 운영자)에서는 **동일 인물**일 수 있다. 그러나 권한·감사·책임 모델 설계에서는 항상 **분리된 두 페르소나**로 다룬다.

---

## 2. P-OP — 플랫폼 운영자

### 정체성
- DevSecOps 엔지니어, SRE, 또는 인프라 관리자
- 본 플랫폼의 **컨트롤 플레인 자체**를 운영
- Proxmox 노드 등록·해제, 플랫폼 업데이트, 백업, 인증서, 모니터링을 책임짐

### 핵심 목표
- 플랫폼의 가용성·무결성 유지
- 사고 발생 시 빠른 인지·복구
- 보안 패치·취약점 대응
- 빌링 데이터의 무결성 보장

### 주요 관심사
- 노드 헬스, 컨트롤 플레인 가용성
- 감사 로그의 완전성과 변조 방지
- 시크릿·키 관리
- 백업·DR 검증

### 대표 시나리오
1. 신규 미니 PC를 Proxmox와 함께 구매해 노드 풀에 등록한다.
2. CI/CD 파이프라인을 통해 컨트롤 플레인을 무중단 업데이트한다.
3. 노드 하트비트 실패 알람을 받고 런북에 따라 조치한다.
4. 빌링 데이터의 머클 트리 루트를 정기 검증한다.

### 권한 윤곽
- 전 자원 read
- 컨트롤 플레인 구성·노드 수명주기 write
- **빌링 데이터 단가 수정 권한은 별도 분리** (SoD)

---

## 3. P-CSP — CSP 사업자

### 정체성
- 본 플랫폼을 설치해 **자기 고객(테넌트)에게 클라우드 서비스를 판매**하는 사업 주체
- 가격표, SLA, 약관, 청구·수금 정책의 최종 결정자

### 핵심 목표
- 매출·수익 가시성 확보
- 가격 정책의 변경·승인 흐름 통제
- 고객 가입·해지·미납 처리
- 규제(전자상거래법, 부가통신사업, 개인정보보호법) 준수

### 주요 관심사
- 가격 정책 변경 이력
- 청구서 무결성과 재현성(idempotency)
- 미납·예산 초과 고객 처리
- 사업적 의사결정에 필요한 비용 분석 보고서

### 대표 시나리오
1. 인스턴스 타입 단가를 인상하고, 변경 이력을 감사 로그에서 확인한다.
2. 월말에 전 고객 청구서를 일괄 발행하고 PDF를 보관한다.
3. 미납 고객에 대해 자원 정지 정책을 적용한다.
4. 분기 비용 분석 보고서로 신규 인스턴스 타입 도입 여부를 결정한다.

### 권한 윤곽
- 가격 정책 read/write (단, 변경은 감사 대상)
- 전 테넌트 빌링 데이터 read
- **테넌트 내부 자원 read 권한은 원칙적으로 없음** (개인정보 보호)

---

## 4. P-TA — 테넌트 관리자

### 정체성
- P-CSP의 고객 조직 안에서 **자원·사용자·예산을 책임지는 사람**
- 사내 개발팀에 VM·스토리지를 발급하고 비용을 관리

### 핵심 목표
- 사내 자원 사용 현황 가시화
- 사내 사용자(P-EU) 권한 부여·회수
- 월 예산 한도 내 비용 통제
- 보안 그룹·네트워크 정책 관리

### 주요 관심사
- 사내 사용자별 사용량
- 예산 알림과 초과 시 동작
- 보안 그룹 변경 이력
- 청구서·세금계산서

### 대표 시나리오
1. 신규 입사자에게 개발자 역할로 사용자 계정을 발급한다.
2. 월 예산 임계치 80% 도달 알람을 받고 미사용 VM을 정리한다.
3. 보안 그룹에 신규 인바운드 규칙을 추가하고 감사 로그에 사유를 남긴다.
4. 월 청구서를 다운로드하고 사내 회계 시스템에 등록한다.

### 권한 윤곽
- 자기 테넌트 내 자원 read/write
- 자기 테넌트 내 사용자·역할 관리
- 자기 테넌트 빌링 데이터 read
- **다른 테넌트의 자원·데이터는 일체 접근 불가** (강제 격리)

---

## 5. P-EU — 엔드 유저

### 정체성
- 테넌트 안에서 **실제로 자원을 만들어 쓰는** 개발자/엔지니어
- 일반적으로 자기 자원만 관리, 비용에는 큰 권한 없음

### 핵심 목표
- 필요한 VM·스토리지·DB를 빠르게 생성
- SSH/콘솔/API로 자원에 접근
- 본인 자원의 성능·상태 모니터링

### 주요 관심사
- 인스턴스 생성 속도와 안정성
- API/CLI 사용성
- 본인 자원의 사용량 가시성

### 대표 시나리오
1. 웹 콘솔에서 Ubuntu VM 4GB를 생성하고 SSH 키를 등록한다.
2. CLI로 본인 VM 목록을 조회하고 스냅샷을 만든다.
3. API 키를 발급해 자동화 스크립트에서 사용한다.
4. 본인의 월 사용량을 대시보드에서 확인한다.

### 권한 윤곽
- 자기가 만든 자원에 한해 read/write
- 같은 테넌트 내 자원 read (역할에 따라 다름)
- **빌링 단가·테넌트 설정·다른 사용자 관리 권한 없음**

## 6. 페르소나 매트릭스 (요약)

| 권한 영역 | P-OP | P-CSP | P-TA | P-EU |
|---|---|---|---|---|
| 노드 등록·해제 | ✅ | ❌ | ❌ | ❌ |
| 가격 정책 수정 | ❌ (read) | ✅ | ❌ | ❌ |
| 전 테넌트 빌링 read | ✅ | ✅ | ❌ | ❌ |
| 자기 테넌트 빌링 read | n/a | n/a | ✅ | 일부 |
| 테넌트 내 자원 CRUD | ❌ | ❌ | ✅ | 본인 자원만 |
| 감사 로그 read | ✅ | ✅ (사업 관련) | ✅ (자기 테넌트) | ❌ |
| 감사 로그 변경 | ❌ | ❌ | ❌ | ❌ |

> 위 표는 **개념 수준 매트릭스**이며, 실제 RBAC 정의는 Phase 2에서 별도 문서로 확정한다.

## 7. 페르소나가 ADR/위협 모델에 영향을 주는 방식

- **위협 모델**: 각 페르소나는 STRIDE의 공격 주체이자 보호 대상이다. 특히 P-EU가 P-TA를, P-TA가 P-CSP를, P-CSP가 P-OP를 권한 상승(Elevation of Privilege) 시도하는 시나리오를 항상 가정한다.
- **감사 로그**: 모든 상태 변경 이벤트는 페르소나 코드(P-OP/P-CSP/P-TA/P-EU)와 함께 기록한다.
- **권한 분리(SoD)**: P-CSP의 가격 정책 수정과 P-OP의 인프라 운영은 **반드시 분리**한다. 1인 운영 환경이라도 두 권한을 동일 자격증명으로 합치지 않는다.
