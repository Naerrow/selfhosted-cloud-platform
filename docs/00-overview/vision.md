# Vision

> 본 문서는 프로젝트가 "왜" 존재하는지를 단 하나의 가치 명제로 못 박는다.
> 모든 기능 추가·삭제 결정은 이 문서로 회귀해 검증한다.

## 1. 한 줄 가치 명제

> 단일 미니 PC부터 다중 서버까지 동일한 콘솔과 빌링 흐름으로 운영하는 **셀프호스팅 클라우드 플랫폼**.
> 단, 기능의 폭보다 **운영·보안·감사 가능성의 깊이**를 우선한다.

## 2. 배경

퍼블릭 클라우드(AWS, Azure 등)는 강력하지만 비용 부담이 크고, 한국에서는 개인정보·금융 데이터의 **데이터 주권** 이슈가 존재한다. OpenStack 같은 풀스택 IaaS는 자체 구축이 가능하지만 **대규모 인프라를 전제**로 하므로 개인·소규모 조직이 다루기에는 진입 장벽이 매우 높다.

한편 Proxmox VE는 가벼운 하이퍼바이저로서 미니 PC에서도 동작하지만, **자체 클러스터링이 저지연 LAN(노드 간 지연 5ms 미만)을 전제로 해 멀티사이트·광역 묶음에 부적합**하고, 본격적 멀티테넌시·빌링이 빠져 있어 그대로 "클라우드 사업"에 사용하기에는 부족하다.

본 프로젝트는 이 간극 — "OpenStack은 너무 크고 Proxmox 단독으로는 부족한 영역" — 을 메우는 **얇은 컨트롤 플레인**을 설계·구현한다.

## 3. 목표

### 3.1 일차 목표 — 이 플랫폼이 해결하는 문제

본 프로젝트는 다음 사용자에게 실질적 가치를 제공하는 것을 일차 목표로 한다.

- **퍼블릭 클라우드의 비용 부담과 데이터 주권 이슈**에서 벗어나 자체 클라우드를 운영하려는 1인 운영자 또는 소규모 조직에게, 미니 PC 1~수 대만으로 시작 가능한 진입점을 제공한다.
- 서버부터 미니 PC까지 **혼합 하드웨어**를 단일 콘솔로 운영할 수 있는 추상화를 제공한다.
- 종량제 청구서가 실제로 떨어지는 **준-CSP(quasi-CSP) 환경**을 제공해, 셀프호스팅 클라우드를 사업적 형태로 운영해 보려는 사용자가 시행착오 비용을 낮출 수 있게 한다.
- 누군가 본 플랫폼을 fork·운영할 때 **무엇을 왜 그렇게 만들었는지** 의사결정 흐름을 그대로 따라갈 수 있게 한다.

### 3.2 설계·운영 원칙 — 깊이 우선

위 일차 목표를 진지하게 추구하려면 **기능 폭의 확장**이 아니라 **운영·보안·감사 가능성의 깊이**가 필요하다고 본다. 따라서 본 프로젝트는 다음 원칙을 타협 없이 적용한다.

- 모든 비자명한 설계 결정은 **ADR**로 근거를 남긴다.
- IaC + CI/CD에 **보안 게이트**(SBOM, 시크릿 스캔, 이미지 스캔, 정책 검사)를 빌드 파이프라인 단계에 내장한다.
- **위협 모델링과 한국 규제 매핑**(CSAP, 개인정보보호법, 전기통신사업법)을 설계 단계에서 통합한다.
- **멀티테넌시 격리**를 컴퓨팅·네트워크·스토리지·메타데이터 네 층에서 엄밀하게 다룬다.
- **빌링은 결제 기능이 아니라 감사 대상 시스템**으로 다룬다. 무결성·감사·권한 분리(SoD)가 핵심.
- **관측 가능성**(logs/metrics/traces)과 SLO 기반 운영 사고를 코드 이전에 정의한다.
- **운영 자동화**(런북, 헬스 체크, 사고 대응)를 문서와 코드 양쪽에 남긴다.

## 4. 성공 기준

본 프로젝트는 다음을 모두 충족할 때 "성공"으로 간주한다.

| 영역 | 기준 |
|---|---|
| 설계 산출물 | ADR 인덱스, C4 다이어그램, 위협 모델, 컴플라이언스 매핑이 모두 리포에 존재하고 상호 참조된다 |
| 동작 산출물 | Phase 1 종료 시점에 VM 생성 → 미터링 → 청구서 PDF의 한 줄기 시연이 가능하다 |
| 운영 산출물 | 각 컴포넌트에 SLO, 런북, 알람이 정의되어 있고, 사고를 가정한 모의 훈련 기록이 1건 이상 존재한다 |
| 보안 산출물 | CI 파이프라인이 SBOM·시크릿 스캔·이미지 취약점 스캔·정책 검사를 게이트로 갖춘다 |
| 온보딩성 | 처음 보는 사람이 README만 보고 짧은 시간 안에 "무엇을, 왜, 어떻게" 답할 수 있다 |

## 5. 비목표 (Non-Goals)

본 프로젝트는 다음을 **명시적으로 추구하지 않는다**. 상세 사유는 [`scope.md`](./scope.md)와 관련 ADR을 참조.

- 실제 상용 CSP와의 기능 동등성 (AWS 클론 완성)
- 다수 퍼블릭 클라우드 동시 지원 (Cloud Bursting은 AWS 1종으로 한정)
- 자율 자동 조치를 수행하는 AI 운영자 (사람 승인 단계를 항상 유지)
- 데스크탑 앱·모바일 네이티브 앱 (PWA 또는 OUT-OF-SCOPE)
- 진짜 PG 실연동 (어댑터 인터페이스 + Mock 구현체까지)
- 빌링의 다중 모델 동시 지원 (종량제 단일 모델만 — ADR-0008)

## 6. 회귀 검증 질문

기능 추가·삭제 결정이 흔들릴 때 다음 4가지를 묻는다.

1. **이 변경이 한 줄 가치 명제에 부합하는가?**
2. **이 변경이 운영·보안·감사 가능성의 깊이를 늘리는가, 아니면 단지 기능 폭을 늘리는가?**
3. **이 변경이 1인 운영 가능 범위 안에 있는가?**
4. **이 변경의 사유가 ADR에 남을 만큼 명료한가?**

세 개 이상에서 "아니오"가 나오면 그 변경은 거부한다.

## 7. 법적 면책

본 문서 및 본 프로젝트 산출물 전반의 한국 규제·법령 인용(CSAP, 개인정보보호법, 전기통신사업법, 전자상거래법 등)은
사전 조사 자료([`docs/06-research/cloud-platforms-and-regulations.md`](../06-research/cloud-platforms-and-regulations.md))의
작성 시점 1차 출처 기반 **일반 정보 제공**이며, **법률 자문이 아니다**.

실제 사업화 의사결정(사업자 등록, 인증 신청, 약관 작성, 개인정보 국외 이전 등)에는
**변호사·관할 기관 검토가 별도로 필요**하다.
