# Scope

> 본 문서는 프로젝트의 IN-SCOPE / OUT-OF-SCOPE를 **사유와 함께** 명시한다.
> "왜 안 만들었는가"는 "무엇을 만들었는가"만큼 설계 의도를 드러낸다.

## 1. IN-SCOPE (단계별)

각 항목의 상세 정의는 [`../../ROADMAP.md`](../../ROADMAP.md) 및 관련 ADR을 참조.

### Phase 0 — Foundation
- 비전·페르소나·용어집·스코프 문서
- C4 Level 1–2 아키텍처 다이어그램
- 핵심 ADR 5~7개
- STRIDE 위협 모델 v0
- CSAP·개인정보보호법 매핑 v0
- 리포 스캐폴딩 + CI/CD 뼈대
- 리스크 레지스터 초안

### Phase 1 — Vertical Slice MVP
- 단일 Proxmox 노드 지원
- VM CRUD, 콘솔 접속(noVNC 프록시)
- 기본 메트릭 수집, 사용 시간 미터링
- 월 청구서 PDF 생성
- 최소 IAM (로컬 사용자 + JWT)
- 웹 콘솔, REST API, CLI
- IaC, CI 파이프라인(린트·테스트·SBOM·시크릿/이미지 스캔), 시크릿 관리

### Phase 2 — IAM & Multi-tenancy
- Domain / Project(Tenant) / User / Role 4계층 RBAC
- OIDC 외부 IdP 연동
- 테넌트별 네트워크/스토리지/쿼터 격리
- 감사 로그(append-only, 변조 탐지)
- API 키 관리

### Phase 3 — Billing Engine
- 미터링 파이프라인
- 가격 정책 엔진
- **종량제 단일 모델**
- 청구서 PDF + 무결성 보장(해시 체인 + 기간 머클 루트 — ADR-0006)
- 빌링 감사 로그
- 실시간 비용 대시보드
- 예산 알림
- PG 어댑터 인터페이스 + Mock 구현체

### Phase 4 — Multi-node Orchestration
- 노드 등록/해제, 헬스 체크
- 규칙 기반 단순 스케줄러
- 페일오버 감지(알림만)
- 자원 풀링 추상화

### Phase 5 — Adjacent Services
- Object Storage (MinIO + S3 호환 API)
- Block Storage (LVM 또는 Ceph 단일 백엔드)
- 보안 그룹, 내부 DNS
- 로드 밸런서 (HAProxy/Nginx)

### Phase 6 — AI Ops (선택)
- 자연어 → 의도 파싱 + 사람 승인 후 실행
- 룰 기반 이상 탐지
- 휴리스틱 비용 최적화 추천
- AI 호출 감사 로그·비용 추적

### Phase 7 — Cloud Bursting (선택)
- AWS EC2/S3 어댑터 1종만
- 임계치 기반 자동 확장
- 비용 합산 빌링 반영

## 2. OUT-OF-SCOPE (명시적 제외)

| 항목 | 결정 | 사유 |
|---|---|---|
| **설치형 데스크탑 클라이언트** (Windows/Mac/Linux 네이티브) | 제외 | 본 프로젝트의 깊이 가치(운영·보안·감사)에 기여하지 않으며, 웹 콘솔로 모든 운영이 가능. |
| **휴대기기 전용 네이티브 클라이언트** (iOS/Android) | 제외 또는 PWA | iOS/Android 양쪽 네이티브는 1인 운영 가능 범위를 크게 벗어남. 알림이 필수면 PWA + Push로 갈음. |
| **승인 없는 운영 자동화** | 제외 | 사전 승인 런북을 사람 승인 모드로만 운영. 자율 자동조치는 운영 위험·검증 비용 모두 크고, 본 프로젝트가 추구하는 **감사 가능성**과도 직접 충돌한다. |
| **여러 퍼블릭 클라우드 동시 확장** | 제외 | AWS 1종만 어댑터 패턴으로 구현. 다중 프로바이더는 어댑터 인터페이스 제시로 갈음. |
| **LLM 백엔드 자동 전환** | 보류 | 핵심 가치 명제와 거리 멈. Phase 6 진입 시 재검토. |
| **Docker 관리형 런타임** | 제외 또는 후순위 | Proxmox는 LXC가 1급 시민. Docker는 사용자 VM 안에서 자유롭게 사용 가능하므로 플랫폼 서비스로 별도 노출할 가치가 낮음. |
| **사용자 이미지 거래·심사 장터** | 제외 | 카탈로그 + 이미지 등록까지만. 사용자 간 거래·심사는 본 프로젝트의 범위를 벗어남. |
| **SLA 위반 자동 보상** | 제외 | 빌링 안정화 후에야 의미 있음. Phase 3 종료 후 재검토. |
| **약정형 인스턴스 요금제** | 제외 | 약정·해지·환불·만료 로직으로 빌링 엔진 복잡도가 2배 이상 됨. 본 프로젝트의 깊이 항목(무결성·감사·SoD)에 기여하지 않음. (ADR-0008) |
| **구독형 정액 과금** | 제외 | 종량제와 산정 방식이 완전히 다른 두 번째 모델. 한 모델을 깊게 다루는 편이 본 프로젝트의 가치 명제에 정합. (ADR-0008) |
| **선불 크레딧 지갑** | 제외 | PG 어댑터 + Mock + 청구서 발행으로 결제 흐름 시연 가능. 크레딧 자체가 불필요해짐. (ADR-0008) |
| **프로모션 할인 체계** | 제외 | 순수 마케팅 기능, 보안·운영·감사 관점의 함의가 거의 없음. (ADR-0008) |
| **운영 PG 직접 연결** (KCP/INICIS/Toss 등) | 제외 | 개인사업자 등록·심사 필요로 데모 환경 부적합. **어댑터 인터페이스 + Mock 구현체**로 갈음. |
| **CSAP 정식 인증 획득** | 제외 | 비용·시간·법인격 요건 모두 본 프로젝트의 운영 가능 범위 밖. **CSAP 통제 항목 매핑 문서**까지만 작성한다. |

## 3. 향후 재검토 항목

다음 항목들은 Phase 진행 중 또는 종료 시 재검토 트리거가 있다.

| 항목 | 재검토 시점 |
|---|---|
| 모바일 PWA 추가 | Phase 5 종료 후 |
| AI 자동 조치 (사람 승인 모드 외) | Phase 6 종료 후, 충분한 감사 로그 축적 시 |
| 두 번째 퍼블릭 클라우드 어댑터 | Phase 7 종료 후 |
| 빌링 다중 모델 (정액·예약) 부활 | 별도 ADR 작성 시에만 |

## 4. 회귀 원칙

새로운 기능 요청이 들어오면 다음 순서로 검토한다.

1. [`vision.md`](./vision.md) 회귀 검증 질문 4종에 답한다.
2. 본 문서의 OUT-OF-SCOPE 사유와 충돌하지 않는지 확인한다.
3. 충돌하면 신규 ADR을 작성해 결정을 뒤집거나, 거부한다.
4. 무관한 신규 기능이면 본 문서에 IN-SCOPE로 추가하고 해당 Phase를 명시한다.
