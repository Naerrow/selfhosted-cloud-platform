# Roadmap

> 본 로드맵은 살아있는 문서다. 매 Phase 종료 시 회고에 따라 갱신한다.
> 시간 추정치는 1인 파트타임 기준 상대적 길이이며, 절대 일정이 아니다.

## 현재 위치

**Phase 0 — Foundation** 진행 중.

## 횡단 원칙 (모든 Phase에 공통 적용)

- 매 Phase는 **새 ADR 1개 이상**, 위협 모델 업데이트, 컴플라이언스 매핑 갱신을 강제한다.
- 매 Phase 종료 시 **데모 GIF + 회고(Retrospective) MD + CHANGELOG 갱신**.
- **첫 공개 커밋 이후** 매주 [`docs/90-meta/decision-log.md`](./docs/90-meta/decision-log.md)에 한 줄이라도 기록한다 (decision-log 기록 정책 본문 참조).
- 비자명한 설계 결정과 외부 효과가 있는 운영 정책은 반드시 ADR 번호로 인용한다.

## Phase 0 — Foundation (코드 0줄, 설계만) · 약 2–3주

이후 모든 단계가 무너지지 않을 토대를 만든다.

**산출물**
- `docs/00-overview/*` 4종 (vision, scope, personas, glossary)
- C4 Level 1–2 아키텍처 다이어그램
- 핵심 ADR (ADR 인덱스 기준; Phase 1 진입 전 ADR-0004~0007 Accepted)
- STRIDE 위협 모델 v0
- CSAP·개인정보보호법 매핑 v0
- 리포 스캐폴딩 + CI/CD 뼈대(린트만 도는 수준)
- 리스크 레지스터 초안

**Exit 기준**
- ADR 인덱스 + C4 다이어그램 + 위협 모델 v0 + CSAP·개인정보보호법 매핑 v0가 리포에 있다.
- **ADR-0004~0007이 Accepted 상태다** (Phase 1 진입 게이트).
- **사전 조사 자료(`docs/06-research/cloud-platforms-and-regulations.md`)의 한국 규제 항목을 Phase 0 종료 시점에 재확인**한다 (R-005 완화책). 변경 사항은 ADR-0010 분류 체계로 처리.
- 처음 보는 사람이 README만 보고 "무엇을, 왜, 어떻게" 답할 수 있다.

## Phase 1 — Vertical Slice MVP · 약 4–6주

VM 한 대를 만들어 사용량을 측정하고 청구서 PDF가 떨어지는 얇은 한 줄기를 완성한다.

**스코프**
- 단일 Proxmox 노드만 지원
- 단일 테넌트 (추상화는 해두되 격리 강제는 약하게)
- 기능: VM CRUD, 콘솔 접속(noVNC 프록시), 기본 메트릭 수집, 사용 시간 미터링, 월 청구서 PDF,
  최소한의 IAM (로컬 사용자 + JWT)
- 클라이언트: **웹 콘솔(최소) + REST API + 간단한 CLI** 3가지만
- DevSecOps 풀스택 적용: IaC, CI 파이프라인(린트/유닛테스트/컨테이너 빌드/SBOM/시크릿 스캔/이미지
  스캔), 시크릿 관리(SOPS 또는 Vault dev), 기본 메트릭/로그 수집

**Exit 기준**
"신규 가입 → VM 생성 → 30분 사용 → 종료 → 한 달치 청구서 PDF 다운로드" 시나리오가 한 번에 시연 가능.
데모 GIF를 README에 등록.

---

## Phase 2 — IAM & Multi-tenancy · 약 3–4주

실제로 "여러 고객"에게 팔 수 있는 형태로 만든다.

**스코프**
- Domain / Project(Tenant) / User / Role 4계층 RBAC
- OIDC 외부 IdP 연동 (예: Keycloak 셀프호스트)
- 테넌트별 네트워크/스토리지/쿼터 격리
- 감사 로그(append-only, 변조 탐지)
- API 키 발급/폐기/스코프 제한

**Exit 기준**
운영자 1명이 고객 2명을 가입시키고, 두 고객의 VM이 네트워크·자원 양쪽에서 격리됨을 시연.
감사 로그가 모든 상태 변경을 캡처.

---

## Phase 3 — Billing Engine · 약 4–5주

빌링이 "장난감"이 아니라 **감사 대상 시스템**임을 보인다.

**스코프 (ADR-0008로 확정)**
- 미터링 파이프라인 (수집 → 정규화 → 시계열 저장)
- 가격 정책 엔진 (인스턴스 타입·스토리지·트래픽 단가)
- **종량제 단일 모델**
- 청구서 생성기 (PDF)
- **청구서 무결성 보장** (해시 체인 + 기간 머클 루트 — ADR-0006)
- 빌링 감사 로그 (단가 변경·수동 조정 전부 추적)
- 실시간 비용 대시보드
- 예산 알림
- PG 어댑터 인터페이스 + Mock 구현체

**제외 (ADR-0008 사유)**: 약정형 인스턴스, 구독형 정액 과금, 크레딧, 쿠폰

**Exit 기준**
1개월치 미터링 → 청구서 생성, 동일 입력으로 동일 청구서 재생산(idempotency) 보장.
빌링 감사 로그가 별도 채널로 보관.

---

## Phase 4 — Multi-node Orchestration · 약 3–4주

여러 Proxmox 노드를 하나의 풀로 운영한다.

**스코프**
- 노드 등록/해제 / 헬스 체크 / 하트비트
- 단순 스케줄러 (자원 여유 기반 — AI 없이 규칙 기반부터)
- 페일오버 감지 (자동 마이그레이션은 아직 X, 알림만)
- 자원 풀링 추상화

**Exit 기준**
미니 PC 2대 + 서버 1대 같은 혼합 환경에서 단일 콘솔로 노드 등록·VM 배치·노드 다운 시 알림 시연.

---

## Phase 5 — Adjacent Services · 약 4–5주

AWS 유사 서비스 라인업을 모듈별로 확장한다.

**스코프**
- Object Storage: MinIO 통합 + S3 호환 API 프록시
- Block Storage: Cinder 유사 추상화 (LVM 또는 Ceph 단일 백엔드)
- 네트워킹: 보안 그룹(방화벽 룰), 내부 DNS
- 로드 밸런서: HAProxy 또는 Nginx 기반

각 서비스마다 ADR + 위협 모델 업데이트 + 빌링 단가 추가.

**Exit 기준**
"S3에 파일 올리고 → VM에서 mount해서 사용 → 트래픽이 미터링·청구에 반영"까지 한 줄기로 시연.

---

## Phase 6 — AI Ops (선택) · 약 2–3주

AI 기능을 "있어 보이는" 게 아니라 **감사 가능한** 형태로 만든다.

**스코프**
- 자연어 → 의도 파싱 → **사람 승인 후** 실행하는 워크플로우(런북 호출)
- 이상 탐지 (룰 기반 + 간단한 통계)
- 비용 최적화 추천 (휴리스틱)
- AI 호출 자체에 대한 감사 로그·비용 추적

**Exit 기준**
"이번 주 비용이 많이 나간 테넌트 알려줘"가 자연어로 동작하고, "VM 종료해줘"는 항상 확인 단계를 거친다.

---

## Phase 7 — Cloud Bursting (선택) · 약 3–4주

퍼블릭 클라우드(AWS) 1종만 우선 연동한다.

**스코프**
- AWS EC2/S3 어댑터
- 정책 기반 트리거 (임계치 초과 시 EC2 인스턴스 발사)
- 비용 합산 (내부 + AWS) 빌링 반영
- 데이터 송신 비용·데이터 주권 관련 운영 가이드 문서

**Exit 기준**
내부 자원 80% 초과 시 AWS EC2 1대가 자동 생성, 청구서에 내부/외부 항목이 합쳐서 표시.

---

## Phase 8 — 클라이언트 확장 (선택, 최후순위)

- CLI 고도화 (자동완성, 프로파일 관리)
- 데스크탑 앱 / 모바일 앱은 **명시적으로 OUT-OF-SCOPE**.
  사유는 [`docs/00-overview/scope.md`](./docs/00-overview/scope.md) 및 별도 ADR로 기록.
  필요 시 모바일은 PWA로 갈음.
